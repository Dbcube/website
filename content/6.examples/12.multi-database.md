Real systems rarely use one database. A common shape: **relational** data
(users, orders) in PostgreSQL, and a **document** store (events, activity logs,
flexible payloads) in MongoDB. DBCube gives you **one API** for both.

## Configure both connections

Each key under `databases` is a connection you address by name.

```ts
// dbcube.config.js
module.exports = function (config) {
  config.set({
    databases: {
      core: {
        type: "postgres",
        config: {
          HOST: process.env.PG_HOST,
          USER: process.env.PG_USER,
          PASSWORD: process.env.PG_PASSWORD,
          DATABASE: process.env.PG_DATABASE,
          PORT: 5432,
        },
      },
      events: {
        type: "mongodb",
        config: {
          HOST: process.env.MONGO_HOST,
          USER: process.env.MONGO_USER,
          PASSWORD: process.env.MONGO_PASSWORD,
          DATABASE: process.env.MONGO_DATABASE,
          PORT: 27017,
        },
      },
    },
  });
};
```

## Use them side by side — same builder, two engines

```ts
import { dbcube } from "dbcube";

const core = dbcube.database("core");      // PostgreSQL
const events = dbcube.database("events");  // MongoDB

interface User { id: number; email: string; name: string; }
interface Event { id: number; type: string; user_id: number; payload: Record<string, unknown>; at: string; }

// Write to Postgres, then record a flexible event document in Mongo — identical API
export async function signUp(email: string, name: string): Promise<User> {
  const [user] = await core.table<User>("users").insert([{ email, name }]);

  await events.table<Event>("events").insert([
    {
      id: Date.now(),                 // Mongo has no autoincrement — supply ids
      type: "user.signed_up",
      user_id: user.id,
      payload: { email, source: "web" },
      at: new Date().toISOString(),
    },
  ]);

  return user;
}
```

## Query each store the same way

```ts
// Relational query (PostgreSQL)
export async function activeUsers() {
  return core.table<User>("users")
    .where("status", "=", "active")
    .orderBy("name", "ASC")
    .get();
}

// Document query (MongoDB) — same where/orderBy/limit chain
export async function recentEvents(userId: number) {
  return events.table<Event>("events")
    .where("user_id", "=", userId)
    .orderBy("at", "DESC")
    .limit(50)
    .get();
}
```

## Transactions are per-database

A transaction is scoped to **one** database (the engine holds the transaction).
Use one where it counts — your relational core — and treat the event store as a
best-effort sink.

```ts
export async function placeOrder(userId: number, totalCents: number) {
  // Atomic on the relational side
  const order = await core.transaction(async (trx) => {
    const [o] = await trx.table("orders").insert([{ user_id: userId, total_cents: totalCents, status: "paid" }]);
    await trx.table("users").where("id", "=", userId).decrement("wallet_cents", totalCents);
    return o;
  });

  // Fire-and-forget analytics event to Mongo (a failure here must not undo the order)
  events.table("events").insert([
    { id: Date.now(), type: "order.placed", user_id: userId, payload: { orderId: order.id }, at: new Date().toISOString() },
  ]).catch((e) => console.error("event log failed:", e));

  return order;
}
```

::callout{type="warning"}
There is no single transaction that spans **two different databases** — that's a
property of databases, not of DBCube (no engine can `COMMIT` another engine's
work). For cross-store consistency use the **outbox / saga** pattern: commit
locally, then publish the event, with compensation on failure.
::

::callout{type="info"}
The same applies to MySQL, SQLite and Turso — add a connection under `databases`
and address it by name. The query builder is identical across every engine; only
engine-specific features (SQL window functions, Mongo command documents) differ,
and those live behind `raw()`.
::
