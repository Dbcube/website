## Philosophy

No synthetic mocks, no cherry-picking. The suite lives in the
[`benchmarks/`](https://github.com/Dbcube) folder and runs DBCube against
**Prisma, Drizzle, TypeORM and Knex** on a **real PostgreSQL 16** database with
identical schema, identical data, identical operations, the same machine and the
same connection budget. Every contender is ~80 lines you can read in
`src/contenders/*.js`.

## Run it in four commands

```bash
cd benchmarks
npm install
npm run db:up          # PostgreSQL 16 + MySQL + Mongo via docker compose
npm run prepare:db     # create + seed the bench database (1,000 users, 5,000 orders)
npx prisma generate    # Prisma needs its generated client
npm run bench          # full run (~3-5 min)   |   npm run bench:quick for a fast pass
```

Results land in two files:

- `results/latest.md` — human-readable table
- `results/latest.json` — raw numbers + environment metadata (Node version, OS,
  library versions, p95, throughput)

::callout{type="info"}
The databases run via `examples/docker-compose.yml` on ports **30045** (MySQL),
**30046** (PostgreSQL) and **30047** (MongoDB). Before a run, kill any stale
engine process and let `prepare-db.js` reset the `bench` database.
::

## What we measure, and why

| Operation | Why it matters |
|---|---|
| SELECT by primary key | The single most common query in any app |
| Filtered list (WHERE + ORDER + LIMIT) | Every list / table view in a product |
| COUNT with filter | Dashboards, pagination totals |
| Relation load (50 users + their orders) | The N+1 killer — batched eager loading |
| INSERT single row | Form submissions, API writes |
| UPDATE by primary key | The standard mutation |
| Transaction (2 atomic updates) | Money movement, inventory — the correctness path |
| Bulk INSERT 1,000 rows | Imports, ETL, seeding |
| 100 concurrent PK lookups | Real traffic is parallel, not sequential |

**Reported metrics**: median latency, p95 latency, throughput (ops/s). Warmup
iterations are excluded; connection pools are hot before timing starts.

## Fairness rules

1. **Same database** — PostgreSQL 16 in Docker (`localhost:30046`), one schema
   from `src/prepare-db.js`. No ORM-specific migrations.
2. **Same data** — 1,000 users + 5,000 orders, seeded identically.
3. **Same connection budget** — every contender gets a pool of ≤ 10.
4. **Same code shape** — each operation is the *idiomatic* form for that ORM
   (`findUnique` for Prisma, `find()` for DBCube, …).
5. **No table-growth bias** — the bulk-insert sink table is truncated outside
   the timed region.
6. **Latest stable versions** — recorded in the results file.

## Reading the numbers honestly

- **DBCube is #1 of all five** for PK reads, updates, transactions and
  concurrent traffic.
- **Knex** — a thin layer over the raw driver — is still the raw-throughput
  champion for single-row bulk and relation loads. DBCube beats Prisma and
  Drizzle there but not Knex.
- The **transaction** win is structural: `db.transaction()` and `db.batch()`
  commit a whole sequence of writes in one network round-trip, while the others
  pay one per statement.
- The **concurrency** result improved dramatically this release — from **3.5×
  behind Prisma to a dead heat**.

::callout{type="warning"}
Run benchmarks on a **quiet machine**. A background compile or Docker build will
inflate p95 across *all* contenders and make the comparison meaningless. Prefer
comparing **deltas within the same run**, not absolute numbers across runs.
::

::callout{type="info"}
Benchmarking against a **remote/cloud** database instead? The network round-trip
will dominate and every ORM will look the same — that's physics, not the engine.
Read [Where speed shows](/performance/where-speed-shows) to measure fairly and
interpret the numbers correctly.
::

## How the runner works

`src/run.js` orchestrates everything:

1. Warms each contender's connection pool (excluded from timing).
2. For each operation, runs N timed iterations per contender, interleaved to
   spread out transient system noise.
3. Computes median / p95 / ops-per-second.
4. Writes `results/latest.md` and `results/latest.json`.

Each contender implements the same nine operations behind a tiny interface, so
adding a new library to the comparison is one file in `src/contenders/`.
