## table:fresh

Drops existing tables and recreates them from your `.table.cube` files.

::callout{type="warning"}
**Destructive Operation**: This command drops all existing tables and recreates them. All data will be lost. Use with caution!
::

### Usage

```bash
dbcube run table:fresh
```

### What It Does

1. Finds all `.table.cube` files in `dbcube/` directory
2. Resolves dependencies between tables (foreign keys)
3. Drops existing tables in reverse dependency order
4. Creates tables from .cube files in correct order
5. Shows progress for each table

### Example Output

```bash
$ dbcube run table:fresh

Reading .table.cube files...
✓ Found 3 table definitions

Resolving dependencies...
✓ Dependency order: users → posts → comments

Dropping existing tables...
✓ Dropped: comments
✓ Dropped: posts
✓ Dropped: users

Creating tables...
✓ Created: users (5 columns)
✓ Created: posts (8 columns, 1 foreign key)
✓ Created: comments (6 columns, 2 foreign keys)

Schema installation complete!
```

### When to Use

- Setting up a new development environment
- Resetting your database to a clean state
- Major schema changes during development
- Testing with a fresh database

### When NOT to Use

- Production environments (data loss!)
- When you have data you want to keep
- For minor schema updates

## table:refresh

Updates existing table structures to match your `.table.cube` files without dropping tables.

### Usage

```bash
dbcube run table:refresh
```

### What It Does

1. Finds all `.table.cube` files
2. Compares with existing database schema
3. Generates ALTER TABLE statements for differences
4. Applies changes while preserving data
5. Shows what was modified

### Example Output

```bash
$ dbcube run table:refresh

Reading .table.cube files...
✓ Found 3 table definitions

Comparing with database schema...

Changes detected:
  users: Add column 'phone'
  posts: Modify column 'content' (text → longtext)
  comments: No changes

Applying schema updates...
✓ users: Added column 'phone' (varchar(20))
✓ posts: Modified column 'content'

Schema refresh complete!
2 tables updated, 1 unchanged
```

### What It Can Do

✓ Add new columns to existing tables
✓ Modify column types (when safe)
✓ Add new tables
✓ Update column constraints
✓ Add indexes

### What It Cannot Do

✗ Remove columns (safety measure)
✗ Rename columns (data loss risk)
✗ Complex type changes (manual migration needed)

### When to Use

- Adding new fields to existing tables
- Updating column types
- Adding new tables to existing database
- Development workflow with existing data
- Production schema updates

### When NOT to Use

- When you need to remove columns (use fresh + backup data)
- For complex schema restructuring
- When renaming tables or columns

## table:alter

Applies explicit migrations from `.alter.cube` files — the precise tool for renames, type changes, and column drops that `table:refresh` won't do automatically.

### Usage

```bash
dbcube run table:alter
```

### What It Does

1. Finds all `.alter.cube` files in `dbcube/` directory
2. Validates each file and its `@database` reference
3. Parses the alter directives and generates dialect-specific statements
4. Executes the changes while preserving existing data

### Supported Directives

| Directive | Change |
|-----------|--------|
| `@changeName` | Rename the table |
| `@addColumn` | Add columns (same syntax as `.table.cube`) |
| `@deleteColumn` | Drop a column |
| `@renameColumn` | Rename a column |
| `@changeType` | Change a column's data type |
| `@changeLength` | Change a column's length |
| `@changeDefault` | Change a default value |
| `@changeOptions` | Add/remove options (`unique`, `not null`, …) |
| `@changeEnumValues` | Replace the values of an enum column |

### Example

```cube
@database("myapp");
@table("users");

@addColumn({
    phone: { type: "varchar"; length: "20"; options: ["null"]; };
});

@changeType({
    bio: {last: "varchar"; new: "text"};
});
```

```bash
$ dbcube run table:alter

🔧 EXECUTING ALTER TABLES — myapp
✓ users
─────────────────────────────
✅ 1 processed · 0 errors
```

See the complete guide: [Alter Tables](/guides/schemas/alter-tables).

## Comparison

| Feature | table:fresh | table:refresh | table:alter |
|---------|-------------|---------------|-------------|
| **Drops tables** | ✓ Yes | ✗ No | ✗ No |
| **Preserves data** | ✗ No | ✓ Yes | ✓ Yes |
| **Adds columns** | ✓ Yes | ✓ Yes | ✓ Yes |
| **Removes columns** | ✓ Yes | ✗ No | ✓ Yes (explicit) |
| **Renames columns** | ✓ Yes | ✗ No | ✓ Yes (explicit) |
| **Changes types** | ✓ Yes | ⚠ When safe | ✓ Yes (explicit) |
| **Safe for production** | ✗ No | ✓ Usually | ✓ Yes (reviewed) |
| **Use case** | Fresh start | Additive updates | Surgical migrations |

## Workflow Examples

### Development Workflow

```bash
# Initial setup
dbcube run database:create
dbcube run table:fresh

# Add columns to users.table.cube
# Update without losing test data
dbcube run table:refresh

# Major refactoring - start fresh
dbcube run table:fresh
dbcube run seeder:add
```

### Production Deployment

```bash
# Initial deployment
dbcube run database:create
dbcube run table:fresh
dbcube run seeder:add
dbcube run trigger:fresh

# Schema update (safe)
dbcube run table:refresh
dbcube run trigger:fresh
```

### Schema Refactoring

```bash
# Save current data first
mysqldump -u root -p myapp_db > backup.sql

# Apply major changes
dbcube run table:fresh
dbcube run seeder:add

# Test thoroughly before deploying
```

## File Discovery

Both commands automatically find `.table.cube` files:

```
dbcube/
├── users.table.cube       ← Found
├── posts.table.cube       ← Found
├── comments.table.cube    ← Found
└── tables/
    └── roles.table.cube   ← Found (subdirectories scanned)
```

## Dependency Resolution

Dbcube automatically resolves table dependencies:

```cube
// users.table.cube (no dependencies)
@database("myapp");
@columns({ id: {...}; name: {...}; });

// posts.table.cube (depends on users)
@database("myapp");
@columns({
  id: {...};
  user_id: {
    type: "int";
    foreign: { table: "users"; column: "id"; };
  };
});

// comments.table.cube (depends on posts and users)
@database("myapp");
@columns({
  id: {...};
  post_id: {
    type: "int";
    foreign: { table: "posts"; column: "id"; };
  };
  user_id: {
    type: "int";
    foreign: { table: "users"; column: "id"; };
  };
});
```

Creation order: **users** → **posts** → **comments**

## Troubleshooting

### Syntax Errors in .cube Files

```bash
$ dbcube run table:fresh

Error: Syntax error in users.table.cube (line 5)
Expected semicolon after type definition
```

**Solution**: Fix the syntax error in the .cube file

### Foreign Key Constraint Errors

```bash
$ dbcube run table:fresh

Error: Cannot create table 'posts'
Foreign key references non-existent table 'users'
```

**Solution**: Check that foreign key references are correct in your .cube files

### Table Already Exists (table:fresh)

```bash
$ dbcube run table:fresh

Error: Cannot drop table 'users' - table is locked
```

**Solution**: Close any database connections, then retry

### Column Type Change Not Allowed (table:refresh)

```bash
$ dbcube run table:refresh

Warning: Cannot change column 'age' from INT to VARCHAR
Manual migration required
```

**Solution**: Use `table:fresh` or write manual migration
