**Dbcube Studio** is a local, self-hosted GUI for your Dbcube project. It runs entirely on your machine and talks only to the databases in your `dbcube.config.js` — nothing leaves your computer.

Think of it as Prisma Studio's bigger sibling: not just a data viewer, but a full control panel for your database — data, **schema**, **queries**, **seeders** and **triggers** — across **PostgreSQL, MySQL, SQLite and MongoDB**.

## Installation

Studio is a separate package you install on its own (it is **not** bundled with the CLI):

```bash
npm install -D @dbcube/studio
```

It reuses the `dbcube` already in your project, so no extra configuration is needed.

## Launch

```bash
npx dbcube studio
```

It starts on **`http://localhost:9944`** (falling back to the next free port if it's busy) and opens your browser automatically.

You can also run it directly (handy if your CLI isn't updated yet):

```bash
npx @dbcube/studio      # or: npx dbcube-studio
```

### Flags

```bash
dbcube studio --port 9944      # preferred port (auto-fallback if taken)
dbcube studio --host 0.0.0.0   # expose on your network
dbcube studio --no-open        # don't open the browser
```

## Features

### Database picker
Switch between every database defined in your `dbcube.config.js` from the top bar. The same UI works for SQL engines and MongoDB.

### Table editor (data + CRUD)
- Paginated data grid with sorting and per-column filters.
- **Insert / edit / delete** rows.
- **Inline cell editing** — double-click a cell to edit it in place (tables with a primary key).
- **Foreign keys as autocomplete** — when a column references another table, editing shows a searchable dropdown of the referenced rows. It searches **any column** (primary key or text), so it works even on tables without a `name` field, and scales to tens of thousands of rows (server-side search).
- **Foreign-key cells** render as a `{ }` chip; click it to see the full referenced row as JSON.
- Auto-generated fields (autoincrement, `uuid`, Mongo `_id`) are hidden when inserting.

### Table definition
See every column with its type, length, primary key, foreign keys, nullability and default.

### Visual schema editor
Create and change tables without writing SQL — Studio generates the exact Dbcube `.cube` grammar and applies it:

- **Create table** — a form (columns, types, options, PK/AI/NN/UQ, defaults, foreign keys) writes a `.table.cube` and applies it.
- **Edit a column** — rename, change type/length/default, add/remove options → generates the matching `.alter.cube` (`@changeType`, `@changeLength`, `@renameColumn`, `@changeOptions`, …), applies it with the schema-engine and records it in `dbcube/migrations.json`.
- **Add column**, **drop column**, **drop table**.

### SQL / Query console
- Run **SQL** (or a **MongoDB command document**) against the selected database.
- **Schema-aware autocomplete**: tables and columns of the current database plus SQL keywords, right at your cursor (Tab/Enter to insert).
- **Editable results** — for single-table `SELECT`s with a primary key, edit result cells inline.
- **Saved queries with groups & subfolders**: organize queries into folders, drag them between groups, rename and delete — stored in your project at `.dbcube/studio/queries.json`.
- Run with **Cmd/Ctrl + Enter**.

### Schema Visualizer
An interactive diagram of your tables with animated relationship edges connecting each **foreign-key column to its referenced primary key**. Drag tables to rearrange — the layout is **saved per database**.

### Seeders
List your `.seeder.cube` files, **view and edit** their rows, **add records**, **create** new seeders from the UI, and **run** them.

### Triggers
Create Dbcube **JavaScript triggers** (`.trigger.cube`) from the UI: pick the table and event (`beforeAdd`, `afterAdd`, `beforeUpdate`, `afterUpdate`, `beforeDelete`, `afterDelete`), write the function body, and Studio generates the file and applies it. Existing trigger files are listed with their handlers.

### Migrations
See applied `.alter.cube` migrations (batch, date, reversibility) and which files are still pending.

## How it works

Studio is a zero-dependency Node HTTP server that:

1. Reads your `dbcube.config.js` to list databases.
2. Uses the `dbcube` ORM in your project for every read and write, so behavior matches your app exactly.
3. Introspects the schema with the same verified queries as `dbcube run pull` and merges the foreign keys declared in your `.cube` files.
4. Serves a single-page app for the UI.

Everything stays on your machine — Studio only connects to the databases in your config.
