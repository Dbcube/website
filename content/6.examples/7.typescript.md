Dbcube is TypeScript-first. Generate interfaces from your schema once, and the
row type flows through every query — typos and wrong types fail at **compile
time**, not in production. (`examples/08-typescript.ts`)

## Generate the types

```bash
npx dbcube generate        # reads your .cube schema → writes dbcube/types.ts
```

Then import what you need:

```ts
import { dbcube } from "dbcube";
import type { User, Order } from "./dbcube/types";
```

## `table<T>()` flows the row type through the whole chain

```ts
const db = dbcube.database("demo");

// → User[]
const users = await db.table<User>("users")
  .where("age", ">", 30)
  .orderBy("age", "DESC")
  .get();

// → User | null
const first = await db.table<User>("users").first();
if (first) console.log(first.name.toUpperCase()); // first.name is string ✅
```

## Inserts are typed too

`insert` takes `Partial<T>`, so generated columns (`id`, `uuid`, defaults) are
optional, but a wrong field name or type is a compile error.

```ts
await db.table<User>("users").insert([
  { name: "Edsger Dijkstra", email: "edsger@example.com", age: 72 },
]);

// await db.table<User>("users").insert([{ nmae: "x" }]); // ❌ TS error: 'nmae' does not exist
```

## Pagination and chunking carry the type

```ts
// PaginatedResult<User>
const page = await db.table<User>("users").paginate(1, 10);
const names: string[] = page.items.map((u) => u.name);

// chunk callbacks receive User[]
await db.table<User>("users").chunk(100, (rows: User[]) => {
  for (const u of rows) console.log(u.email);
});
```

## Type ad-hoc projections with `raw<R>()`

```ts
const stats = await db.raw<{ status: string; n: number }>(
  "SELECT status, COUNT(*) AS n FROM users GROUP BY status"
);
console.log(stats[0].n + 1); // n is number ✅
```

## What the compiler catches for you

```ts
// ❌ wrong field name
await db.table<User>("users").insert([{ nmae: "x" }]);

// ❌ wrong type out
const bad: number = (await db.table<User>("users").first())!.name;
```

::callout{type="success"}
Re-run `npx dbcube generate` whenever your `.cube` schema changes — the types
stay in lock-step with the database, so a schema change that breaks your code
surfaces as a red squiggle, not a runtime 500.
::
