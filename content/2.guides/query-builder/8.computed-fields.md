Computed fields are **virtual columns** that don't exist in the database. They are declared once in your `.table.cube` file with a JavaScript function and calculated automatically every time you select them — Dbcube fetches the real columns they depend on, runs your function per row, and returns the final value.

<!-- DIAGRAM: computed-fields-flow (see diagrams.md) -->

## Declaring a Computed Field

In your `.table.cube` file, give the column a `value` with `@compute` and reference real columns with `@column`:

```ts
@database("myapp");

@meta({
    name: "users";
    description: "User accounts";
});

@columns({
    id: {
        type: "int";
        options: ["primary", "autoincrement"];
    };
    name: {
        type: "varchar";
        length: "255";
        options: ["not null"];
    };
    last_name: {
        type: "varchar";
        length: "255";
        options: ["not null"];
    };
    full_name: {
        type: "string";
        value: @compute(() => {
            const name = @column(name) + " " + @column(last_name);
            return name;
        });
    };
});
```

When you run `dbcube run table:fresh` (or `table:refresh`), the computed field is registered in the local config database — the `full_name` column is **not** created in your actual table.

## Querying Computed Fields

Enable computed fields on the connection with `useComputes()`:

```typescript
import { Dbcube } from 'dbcube';

const dbcube = new Dbcube();
dbcube.init();

// Enable computed fields for this connection
const db = await dbcube.database('myapp').useComputes();

const users = await db.table('users')
  .select(['id', 'full_name'])
  .get();

console.log(users);
// [{ id: 1, full_name: 'Albert Araya' }, { id: 2, full_name: 'Omar Araya' }]
```

### What Happens Under the Hood

1. Dbcube detects that `full_name` is a computed field.
2. It extracts the **dependencies** from your function (`name`, `last_name`) and adds them to the real SELECT.
3. The query engine fetches `id`, `name`, `last_name` from the database.
4. Your compute function runs for every row with the real values substituted into each `@column(...)` reference.
5. The dependency columns you didn't ask for are stripped, so you receive exactly `id` and `full_name`.

::callout{type="info"}
**Zero storage, always fresh.** Because the value is computed at read time, it can never go stale — unlike denormalized columns that need triggers or batch jobs to stay in sync.
::

## Type Conversion

The `type` you declare controls the output conversion:

| Declared type | Conversion applied |
|---------------|-------------------|
| `string` | `String(result)` |
| `number` | `Number(result)` (falls back to `0` if `NaN`) |
| `boolean` | `Boolean(result)` |
| `date` | `new Date(result)` if not already a `Date` |
| `object` | Returned as-is |

## More Examples

### Price with Tax

```ts
price_with_tax: {
    type: "number";
    value: @compute(() => {
        return Math.round(@column(price) * 1.19 * 100) / 100;
    });
};
```

### Age from Birth Date

```ts
age: {
    type: "number";
    value: @compute(() => {
        const birth = new Date(@column(birth_date));
        const diff = Date.now() - birth.getTime();
        return Math.floor(diff / (365.25 * 24 * 60 * 60 * 1000));
    });
};
```

### Status Label

```ts
status_label: {
    type: "string";
    value: @compute(() => {
        const map = { 1: "Active", 2: "Suspended", 3: "Deleted" };
        return map[@column(status)] ?? "Unknown";
    });
};
```

## Behavior Notes

- **Null safety**: if a dependency is `NULL`, `@column(...)` resolves to `null` inside your function — handle it explicitly when needed.
- **Errors don't break queries**: if a compute function throws, the field resolves to `null` for that row and the error is logged.
- **Per-connection opt-in**: `useComputes()` returns a *new* `Database` instance with computes enabled; connections created without it skip the processing entirely (zero overhead).
- **Selecting only real columns**: if your `select()` doesn't include any computed field, no compute processing happens.

## Best Practices

1. **Keep functions pure and fast** — they run once per row in your Node.js process. Avoid I/O inside `@compute`.
2. **Don't compute what SQL does better** — aggregations across rows belong in `groupBy()`/`sum()`, not computed fields.
3. **Name dependencies explicitly** — every real column the function needs must be referenced via `@column(...)` so Dbcube can fetch it.
4. **Use the right `type`** — the conversion guards your API consumers against surprises.
