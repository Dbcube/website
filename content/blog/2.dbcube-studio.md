Great data tools shouldn't make you choose between *seeing* your database and
*shaping* it. Most GUIs are read-mostly viewers; most schema work still happens in
migration files and a terminal. **Dbcube Studio** collapses that split into a
single local app.

```bash
npx dbcube studio
```

That one command opens a full control panel for your project at
`http://localhost:9944` — data, schema, queries, seeders and triggers — across
**PostgreSQL, MySQL, SQLite and MongoDB**. No account, no cloud, no telemetry. It
talks to your databases through the same `dbcube` ORM your app already uses, so
what you see in Studio is exactly what your code sees.

Here's the tour.

## One UI for every engine

Pick any database from your `dbcube.config.js` in the top bar. SQLite, MySQL,
PostgreSQL, MongoDB — the interface is identical, because Studio drives each one
through the ORM rather than a per-engine client.

![Database picker across SQLite, MySQL, PostgreSQL and MongoDB](/studio/database-picker.png)

The **Migrations** panel next to it shows your applied `.alter.cube` history and
anything still pending, so the state of a database is never a mystery.

## Editing that respects your schema

The data grid is fast and familiar: pagination, sorting, per-column filters,
insert/edit/delete — and **double-click any cell** to edit it in place.

![The data grid with inline editing](/studio/data-grid.png)

The part people love: **foreign keys become searchable selects.** When a column
points at another table, you don't type a raw id — you pick the referenced row
from a dropdown that searches *any* column and scales to tens of thousands of rows
with server-side search. Auto-generated fields (autoincrement, `uuid`, Mongo
`_id`) are hidden on insert, so you only fill in what matters.

## Design schema without leaving the GUI

This is where Studio goes beyond a viewer. The **Definition** tab shows every
column — type, constraints, nullability, default — and lets you change the schema
visually.

![The visual schema editor](/studio/table-definition.png)

Create a table from a form and Studio writes a real `.table.cube` and applies it.
Rename a column, change its type or length, add or drop options, and Studio
generates the matching **`.alter.cube`** migration, applies it with the Rust
schema-engine, and records it in `dbcube/migrations.json`.

That last part is the important bit: **every change is a versionable `.cube`
file**, organized under `dbcube/schema/`. The schema you shape by clicking is the
same schema your team reviews in a pull request. No drift between "what the GUI
did" and "what's in git."

## See how it all connects

Foreign keys are easy to declare and hard to keep in your head. The **Schema
Visualizer** draws your tables as draggable nodes with animated edges linking each
foreign-key column to the primary key it references — and it remembers your layout
per database.

![Schema visualizer with animated foreign-key edges](/studio/schema-visualizer.png)

## A real query console

Sometimes you just want to run SQL. Studio's console has **schema-aware
autocomplete** — the tables and columns of the current database plus SQL keywords,
right at your cursor — and for single-table `SELECT`s with a primary key, the
**results are editable inline**.

![The SQL / Query console with editable results](/studio/query-console.png)

Save the queries you reuse, organize them into folders and subfolders, and run
with **Cmd/Ctrl + Enter**. It works with MongoDB too — send a command document and
read the results in the same grid.

## Seed data and write triggers — visually

Seeders shouldn't require hand-writing datasets. Studio lists your `.seeder.cube`
files and lets you edit rows in a spreadsheet-like form, add records, create new
seeders, delete them, and run them.

![The seeder editor](/studio/seeder-editor.png)

And triggers — Dbcube's JavaScript hooks that run on writes — get a proper editor.
Pick a table and event (`beforeAdd`, `afterAdd`, `beforeUpdate`, …), then write the
function body with TypeScript-aware hints. The signature is documented right there:

![The trigger function editor](/studio/trigger-editor.png)

```ts
({ db, oldData, newData, gdb }) => {
  // db      → query builder for THIS table's database
  // oldData → row BEFORE the change (null on insert)
  // newData → row AFTER the change (null on delete)
  // gdb     → reach ANY database: gdb.database("...").table("...")
}
```

Studio writes the `.trigger.cube`, applies it, lists existing triggers with a
delete action, and guards saving against double-clicks so you never end up with a
duplicate.

## Why it's built this way

Three decisions make Studio different from a generic database GUI:

- **It uses your ORM, not a raw driver.** Every read and write goes through the
  same `dbcube` in your project, so foreign keys, computed fields and engine
  quirks behave exactly like they do in your app.
- **The schema is code.** Anything you design in the UI lands as `.cube` files and
  `migrations.json` — reviewable, versionable, reproducible.
- **It's yours, locally.** A zero-dependency Node server that only connects to the
  databases in your config. Nothing leaves your machine.

## Try it

If you already have a Dbcube project:

```bash
npm install -D @dbcube/studio
npx dbcube studio
```

New to Dbcube? Scaffold a project first with `npx dbcube init`, then launch Studio
and watch your schema, data and relationships come to life in one window.
