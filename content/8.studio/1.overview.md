::callout{type="info"}
**One command, your whole database.** `npx dbcube studio` opens a full control
panel for your project — data, schema, queries, seeders and triggers — across
**PostgreSQL, MySQL, SQLite and MongoDB**. It runs 100% on your machine.
::

**Dbcube Studio** is a local, self-hosted GUI for your Dbcube project. Think of it
as Prisma Studio's bigger sibling: not just a data viewer, but a place to *edit
data, design schema, run SQL or Mongo, seed test data, write triggers and track
migrations* — all talking through the same `dbcube` ORM your app uses, so what you
see is exactly what your code sees.

![Dbcube Studio — the data grid](/studio/data-grid.png)

```bash
npx dbcube studio
```

It starts on **`http://localhost:9944`** (falling back to the next free port) and
opens your browser automatically. No account, no cloud, no telemetry.

## Switch database — every engine, one UI

Pick any database defined in your `dbcube.config.js` from the top bar. The exact
same interface drives **SQLite, MySQL, PostgreSQL and MongoDB** — Studio speaks to
each through the Dbcube ORM, so behavior matches your app on every engine.

![Database picker and migrations panel](/studio/database-picker.png)

The **Migrations** panel shows your applied `.alter.cube` history and whatever is
still pending, so you always know the state of a database at a glance.

## Browse and edit data

A fast, paginated data grid with sorting and per-column filters. **Insert, edit
and delete** rows, or **double-click any cell** to edit it in place (on tables
with a primary key).

![Table editor with inline cell editing](/studio/data-grid.png)

::callout{type="success"}
**Foreign keys become searchable selects.** When a column references another
table, editing it shows a dropdown of the referenced rows. It searches **any**
column (primary key or text), so it works even without a `name` field, and scales
to tens of thousands of rows with server-side search. Auto-generated fields
(autoincrement, `uuid`, Mongo `_id`) are hidden when inserting.
::

## Design your schema — no SQL

The **Definition** tab shows every column with its type, constraints, nullability
and default. From here you edit the schema visually and Studio generates the exact
Dbcube `.cube` grammar and applies it with the schema-engine.

![Table definition and column editor](/studio/table-definition.png)

- **Create table** — a form (columns, types, options, PK/AI/NN/UQ, defaults,
  foreign keys) writes a `.table.cube` to `dbcube/schema/tables/` and applies it.
- **Edit a column** — rename, change type/length/default, add or remove options →
  generates the matching `.alter.cube` in `dbcube/schema/alters/`
  (`@changeType`, `@changeLength`, `@renameColumn`, `@changeOptions`, …), applies
  it and records it in `dbcube/migrations.json`.
- **Add column**, **drop column**, **drop table** — all one click.

::callout{type="info"}
Every change is a real, versionable `.cube` file — so the schema you shape in the
UI is the same schema your team reviews in a pull request.
::

## See how it all connects

The **Schema Visualizer** draws your tables as draggable nodes with **animated
edges** linking each foreign-key column to the primary key it references. Rearrange
the layout and it's **saved per database**.

![Schema visualizer with foreign-key relationships](/studio/schema-visualizer.png)

## Run SQL and Mongo — with saved queries

Run **SQL** (or a **MongoDB command document**) against the selected database. The
editor has **schema-aware autocomplete** — tables and columns of the current
database plus SQL keywords, right at your cursor.

![SQL / Query console with editable results](/studio/query-console.png)

- **Editable results** — for single-table `SELECT`s with a primary key, edit the
  result cells inline.
- **Saved queries with groups & subfolders** — organize queries into folders, drag
  them between groups, rename and delete.
- Run with **Cmd/Ctrl + Enter**.

## Seed test data visually

List your `.seeder.cube` files, **view and edit** their rows in a spreadsheet-like
form, **add records**, **create** new seeders, **delete** them, and **run** them —
without hand-writing a single dataset.

![Seeder editor](/studio/seeder-editor.png)

## Write triggers with a real editor

Create Dbcube **JavaScript triggers** from the UI: pick the table and event
(`beforeAdd`, `afterAdd`, `beforeUpdate`, `afterUpdate`, `beforeDelete`,
`afterDelete`), then write the function body in a CodeMirror editor with
TypeScript-aware hints. Studio generates the `.trigger.cube` in
`dbcube/schema/triggers/` and applies it.

![Trigger function editor](/studio/trigger-editor.png)

The handler signature is documented right in the editor:

```ts
({ db, oldData, newData, gdb }) => {
  // db      → query builder for THIS table's database
  // oldData → row BEFORE the change (null on insert)
  // newData → row AFTER the change (null on delete)
  // gdb     → master instance to reach ANY database: gdb.database("...").table("...")
}
```

Existing triggers are listed with their handlers and a **delete** action, and
saving is guarded against double-clicks so a trigger is never created twice.

## Install & launch

Studio is a separate package you install on its own (it is **not** bundled with
the CLI):

```bash
npm install -D @dbcube/studio
npx dbcube studio
```

It reuses the `dbcube` already in your project, so there's nothing else to
configure. You can also run it directly:

```bash
npx @dbcube/studio      # or: npx dbcube-studio
```

### Flags

```bash
dbcube studio --port 9944      # preferred port (auto-fallback if taken)
dbcube studio --host 0.0.0.0   # expose on your network
dbcube studio --no-open        # don't open the browser
```

## How it works

Studio is a zero-dependency Node HTTP server that:

1. Reads your `dbcube.config.js` to list databases.
2. Uses the `dbcube` ORM in your project for every read and write, so behavior
   matches your app exactly.
3. Introspects the schema with the same verified queries as `dbcube run pull` and
   merges the foreign keys declared in your `.cube` files.
4. Serves a single-page app for the UI.

::callout{type="warning"}
**Nothing leaves your machine.** Studio only connects to the databases in your
config — no account, no cloud, no telemetry.
::
