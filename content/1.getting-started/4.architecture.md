Dbcube is not a single library — it is a complete ecosystem where every layer has a single responsibility. The JavaScript/TypeScript packages give you a great developer experience, while the heavy lifting (parsing, query generation, and execution) is delegated to native **Rust engines** for maximum performance.

<!-- DIAGRAM: architecture-overview (see diagrams.md) -->

## The Layers

### 1. Your Application

Your Node.js application talks to Dbcube through one of two entry points:

- **`dbcube` (ORM)** — The all-in-one package. Loads `dbcube.config.js`, registers every configured database, and exposes `dbcube.database('name')` connections.
- **`@dbcube/query-builder`** — The standalone fluent query builder if you only need DML operations.

### 2. The Builders (TypeScript)

| Package | Responsibility |
|---------|---------------|
| `@dbcube/query-builder` | Builds DML operations (SELECT, INSERT, UPDATE, DELETE) as a serializable query plan and orchestrates triggers and computed fields |
| `@dbcube/schema-builder` | Reads `.cube` files, validates them (`CubeValidator`), resolves table dependencies (`DependencyResolver`), and drives schema operations |
| `@dbcube/core` | Shared foundation: engine process management, binary downloads, configuration loading, computed-field and trigger processors, logging |

The builders never generate SQL themselves. They produce a **DML/DDL JSON plan** and hand it to the engines.

### 3. The Rust Engines (Native Binaries)

| Engine | Purpose |
|--------|---------|
| **query-engine** | Receives DML JSON plans, generates dialect-specific queries, executes them against MySQL, PostgreSQL, SQLite, or MongoDB, and returns JSON results |
| **schema-engine** | Parses `.table.cube`, `.alter.cube`, `.seeder.cube`, and `.trigger.cube` files; generates and executes DDL; processes seeders and trigger registration |
| **sqlite-engine** | Dedicated SQLite executor used for embedded/local workflows |

Each engine is a standalone executable compiled for Windows, Linux, and macOS (x64 and ARM64). They communicate using a `PROCESS_RESPONSE:{...}` JSON protocol with HTTP-style status codes (`200` success, `500` error, `600` warning).

### Daemon Mode: Sub-Millisecond Queries

For queries, the engine doesn't spawn per operation — it runs as a **persistent TCP server** with warm connection pools:

1. The first query starts the daemon (`query-engine --action server`) and discovers its port via `.dbcube/daemon/<database>.json`.
2. Every subsequent query travels over the local socket — no process spawn, no reconnection, no handshake.
3. The daemon also powers **transactions** (`begin`/`commit`/`rollback` live on a dedicated pooled connection) and **raw SQL** with bound parameters.

Measured impact: ~0.3ms per query versus 30–80ms with per-query process spawning — the difference between "fast enough" and the fastest path available to Node.js.

Set `DBCUBE_DAEMON=0` to disable daemon mode (operations fall back to one-shot spawn; transactions require the daemon).

::callout{type="info"}
**Why Rust?** Parsing schemas, diffing structures, generating SQL, and pooling connections are CPU- and I/O-bound tasks. Doing them in a compiled native binary keeps the Node.js event loop free and makes operations consistently fast regardless of project size.
::

### 4. The Databases

The same API works across all supported engines:

- **MySQL** — full SQL support
- **PostgreSQL** — full SQL support
- **SQLite** — file-based, stored under `.dbcube/` in your project
- **MongoDB** — the fluent API is translated into native MongoDB operations

## How a Query Flows

<!-- DIAGRAM: query-flow (see diagrams.md) -->

When you run:

```typescript
const users = await db.table('users').where('age', '>', 25).get();
```

1. The **query builder** clones the immutable query state and assembles a DML JSON plan: `{ type: 'select', table: 'users', where: [...] }`.
2. If computed fields are enabled, their **column dependencies** are injected into the SELECT automatically.
3. `@dbcube/core` spawns the **query-engine** binary with the plan and your connection arguments.
4. The engine generates the dialect-specific query, executes it, and prints `PROCESS_RESPONSE:{"status":200,"data":[...]}`.
5. The builder parses the response, applies **computed fields** in-memory, strips the helper dependency columns, and returns plain JavaScript objects.

If triggers are enabled, `beforeAdd`/`afterAdd` (and the update/delete equivalents) wrap step 3–4 for every record, with commit/discard semantics.

## How a Schema Migration Flows

<!-- DIAGRAM: schema-flow (see diagrams.md) -->

When you run `dbcube run table:fresh`:

1. The **schema-builder** scans `dbcube/` recursively for `.table.cube` files.
2. Every file is validated (`CubeValidator`): syntax, types, options, foreign keys, and the `@database` reference against `dbcube.config.js`.
3. The **DependencyResolver** orders the tables so foreign-key targets are created first. If a dependency fails, dependent tables are skipped with a clear error.
4. For each file the **schema-engine** runs three actions: `parse_table` → `generate` → `execute`.
5. The resulting CREATE statement is stored in the local **config.db** (`dbcube_schemas_config`), which is what makes `table:refresh` diffs possible later.

## Local State: the `config.db`

Dbcube keeps a small SQLite database in `.dbcube/config.db` with three tables:

| Table | What it stores |
|-------|----------------|
| `dbcube_schemas_config` | The last applied CREATE statement per table — used by `table:refresh` to compute ALTER diffs |
| `dbcube_computes_config` | Registered computed fields (`@compute` columns) per database |
| `dbcube_triggers_config` | Registered triggers per database, compiled from `.trigger.cube` files |

::callout{type="warning"}
The `.dbcube/` directory is machine-local state. Add it to `.gitignore` — your `.cube` files are the source of truth, not the config database.
::

## Binary Management

The engines are downloaded on demand and cached. You manage them with the CLI:

```bash
# Check for updates and upgrade all binaries
npx dbcube update

# Download a specific engine
npx dbcube run download query-engine

# Pin an exact version
npx dbcube run download query-engine v3.1.1
```

`@dbcube/core` automatically detects your OS and architecture and selects the right binary (`query-engine-windows-x64.exe`, `query-engine-linux-x64`, etc.).

## Package Map

<!-- DIAGRAM: package-dependencies (see diagrams.md) -->

```
dbcube (ORM)
├── @dbcube/query-builder ──┐
│                           ├── @dbcube/core ── Rust engines
@dbcube/schema-builder ─────┘        │
                                     └── query-engine / schema-engine / sqlite-engine
@dbcube/cli  (uses schema-builder + core)
```

| Package | npm | Role |
|---------|-----|------|
| `dbcube` | ORM entry point | Config loading + database registry |
| `@dbcube/query-builder` | Library | Fluent DML API |
| `@dbcube/schema-builder` | Library | `.cube` processing |
| `@dbcube/core` | Library | Engines, processors, config |
| `@dbcube/cli` | CLI | `dbcube` terminal commands |
