## Query Execution Methods

### get() — Retrieve Multiple Records

Executes the query and returns **all** matching rows as an array:

```typescript
const users = await db.table('users')
  .where('status', '=', 'active')
  .get();

console.log(users);
// [{ id: 1, name: 'John' }, { id: 2, name: 'Jane' }, ...]

// Empty result = empty array (never null)
const none = await db.table('users').where('id', '=', -1).get();
console.log(none); // []
```

### first() — Get the First Record

Returns the first matching row, or `null` if nothing matches. Internally applies `LIMIT 1`:

```typescript
const user = await db.table('users')
  .where('email', '=', 'john@example.com')
  .first();

if (user) {
  console.log(user.name);
} else {
  console.log('Not found');
}
```

### find() — Search by Column Value

Shorthand for "first row where column = value". The column defaults to `id`:

```typescript
// By primary key
const user = await db.table('users').find(42);

// By any column
const user = await db.table('users').find('john@example.com', 'email');
```

Returns the record or `null`.

## Aggregation Methods

All aggregations **execute immediately** and return a `Promise<number>` — they are terminal operations, not chain links:

### count()

```typescript
const total = await db.table('users').count();

const active = await db.table('users')
  .where('status', '=', 'active')
  .count();

// Count a specific column (ignores NULLs in that column)
const withEmail = await db.table('users').count('email');
```

::callout{type="info"}
`count()` ignores any `limit()`/`page()` in the chain — it always counts the full matching set, which is exactly what pagination UIs need.
::

### sum() / avg() / max() / min()

```typescript
const revenue = await db.table('orders')
  .where('status', '=', 'paid')
  .sum('amount');

const avgAge = await db.table('users').avg('age');

const newest = await db.table('posts').max('created_at');

const cheapest = await db.table('products')
  .where('stock', '>', 0)
  .min('price');
```

If no rows match, aggregations return `0`.

```typescript
// ✗ Wrong — there is nothing to chain after an aggregation
const n = await db.table('users').count().first();

// ✓ Correct
const n = await db.table('users').count();
```

## Pagination and Limiting

### limit()

```typescript
const latest = await db.table('posts')
  .orderBy('created_at', 'DESC')
  .limit(5)
  .get();
```

### page()

`page(n)` computes the offset from the current limit: `offset = (n - 1) * limit`. **Set `limit()` first** — without it, `page()` has no effect:

```typescript
const pageSize = 20;

const page1 = await db.table('products').limit(pageSize).page(1).get();
const page2 = await db.table('products').limit(pageSize).page(2).get();
```

### paginate() — One-Call Pagination

`paginate(page, perPage)` runs the page query and the total count (same conditions) and returns everything a pagination UI needs:

```typescript
const result = await db.table('products')
  .where('published', '=', true)
  .orderBy('id', 'ASC')
  .paginate(2, 25);

// {
//   items: [...25 rows...],
//   page: 2,
//   perPage: 25,
//   total: 137,
//   totalPages: 6,
//   hasNext: true,
//   hasPrev: true
// }
```

Both queries run in parallel internally, and `count()` ignores limit/page so the total is always the full filtered set.

## Practical Examples

### Dashboard Statistics

```typescript
const [totalUsers, activeUsers, totalRevenue, avgOrder] = await Promise.all([
  db.table('users').count(),
  db.table('users').where('status', '=', 'active').count(),
  db.table('orders').where('status', '=', 'paid').sum('amount'),
  db.table('orders').where('status', '=', 'paid').avg('amount'),
]);
```

### Autocomplete Search

```typescript
async function searchUsers(term: string) {
  return db.table('users')
    .select(['id', 'name', 'email'])
    .where('name', 'LIKE', `${term}%`)
    .orderBy('name', 'ASC')
    .limit(10)
    .get();
}
```

### chunk() — Processing Large Datasets in Batches

`chunk(size, callback)` iterates the full matching set in batches, keeping memory flat. Return `false` from the callback to stop early:

```typescript
await db.table('logs')
  .where('level', '=', 'error')
  .orderBy('id', 'ASC')
  .chunk(500, async (rows, page) => {
    await processBatch(rows);       // 500 rows at a time
    if (page >= 100) return false;  // optional early exit
  });
```

### exists() — Cheap Existence Check

`exists()` selects a constant with `LIMIT 1` — far cheaper than fetching a row:

```typescript
const taken = await db.table('users')
  .where('email', '=', email)
  .exists();   // true | false

if (taken) throw new Error('Email already registered');
```

## Optimization and Best Practices

### 1. Select Only Necessary Columns

```typescript
const users = await db.table('users').select(['id', 'name']).get();
```

### 2. Use count() Instead of get().length

```typescript
// ✗ Fetches every row just to count
const n = (await db.table('users').get()).length;

// ✓ Counts in the database
const n = await db.table('users').count();
```

### 3. Always Order Paginated Queries

Without `orderBy()`, page boundaries are unstable across requests:

```typescript
const rows = await db.table('products')
  .orderBy('id', 'ASC')      // deterministic pages
  .limit(20)
  .page(3)
  .get();
```

### 4. Parallelize Independent Queries

Independent reads can run concurrently with `Promise.all` (see the dashboard example above).
