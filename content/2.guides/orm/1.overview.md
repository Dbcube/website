The `dbcube` package is the recommended way to use Dbcube in an application. It wraps the query builder, loads your configuration, and manages every database connection from a single place.

## Installation

::code-group

```bash [npm]
npm install dbcube
```

```bash [yarn]
yarn add dbcube
```

```bash [pnpm]
pnpm add dbcube
```

```bash [bun]
bun add dbcube
```

::

## Quick Start

```typescript
import { Dbcube } from 'dbcube';

const dbcube = new Dbcube();
dbcube.init();

// Get a connection for any database defined in dbcube.config.js
const db = dbcube.database('myapp');

const users = await db.table('users')
  .where('status', '=', 'active')
  .get();
```

::callout{type="info"}
`Dbcube` implements the **Singleton pattern** â€” every `new Dbcube()` call returns the same instance, so you can safely instantiate it anywhere in your codebase without creating duplicate connection registries.
::

## Configuration: `dbcube.config.js`

The ORM loads `dbcube.config.js` from your project root. It must export a function that receives the config object:

```javascript
require('dotenv').config({ quiet: true });

module.exports = function (config) {
  config.set({
    databases: {
      // SQLite â€” file stored under .dbcube/
      local: {
        type: "sqlite",
        config: {
          DATABASE: process.env.DBCUBE_LOCAL_DATABASE
        }
      },

      // MySQL
      myapp: {
        type: "mysql",
        config: {
          HOST: process.env.DB_HOST,
          PORT: process.env.DB_PORT,
          USER: process.env.DB_USER,
          PASSWORD: process.env.DB_PASSWORD,
          DATABASE: process.env.DB_NAME
        }
      },

      // PostgreSQL
      analytics: {
        type: "postgres",
        config: {
          HOST: process.env.PG_HOST,
          PORT: process.env.PG_PORT,
          USER: process.env.PG_USER,
          PASSWORD: process.env.PG_PASSWORD,
          DATABASE: process.env.PG_NAME
        }
      },

      // MongoDB
      events: {
        type: "mongodb",
        config: {
          HOST: process.env.MONGO_HOST,
          PORT: process.env.MONGO_PORT,
          USER: process.env.MONGO_USER,
          PASSWORD: process.env.MONGO_PASSWORD,
          DATABASE: process.env.MONGO_NAME
        }
      }
    }
  });
};
```

### Connection Parameters

| Parameter | Required for | Description |
|-----------|--------------|-------------|
| `type` | All | `mysql`, `postgres`, `sqlite`, or `mongodb` |
| `DATABASE` | All | Database name (for SQLite, the file name without `.db`) |
| `HOST` | mysql, postgres, mongodb | Server host |
| `PORT` | mysql, postgres, mongodb | Server port |
| `USER` | mysql, postgres, mongodb | Username |
| `PASSWORD` | mysql, postgres, mongodb | Password |

::callout{type="warning"}
**Never hardcode credentials.** Use a `.env` file with `dotenv` as shown above, and keep `.env` out of version control.
::

## API Reference

### `new Dbcube()`

Creates (or returns) the singleton instance.

### `dbcube.init(configCreate?)`

Loads `dbcube.config.js`, registers a `Database` instance for every configured database, and clears the require cache so config changes are always picked up.

```typescript
// Standard initialization
dbcube.init();

// Also register a database that is being created on the fly
dbcube.init({ databaseName: 'newapp', motor: 'mysql' });
```

### `dbcube.database(name)`

Returns the `Database` (query builder) instance for a configured database. From here the entire [Query Builder API](/guides/query-builder/database) is available:

```typescript
const db = dbcube.database('myapp');

await db.table('orders')
  .select(['id', 'total'])
  .where('status', '=', 'paid')
  .orderBy('created_at', 'DESC')
  .limit(20)
  .get();
```

## Working with Multiple Databases

Because every configured database gets its own instance, cross-database workflows are trivial:

```typescript
const mysql = dbcube.database('myapp');      // MySQL
const mongo = dbcube.database('events');     // MongoDB

const user = await mysql.table('users').find(42);

await mongo.table('activity_log').insert([{
  user_id: user.id,
  action: 'login',
  at: new Date()
}]);
```

The same fluent API works on both â€” Dbcube translates it to SQL or native MongoDB operations under the hood.

## Enabling Advanced Features

Computed fields and triggers are opt-in per connection:

```typescript
// Computed fields defined with @compute in your .cube files
const dbWithComputes = await dbcube.database('myapp').useComputes();

// Triggers defined in .trigger.cube files
const dbWithTriggers = await dbcube.database('myapp').useTriggers();
```

See the dedicated guides: [Computed Fields](/guides/query-builder/computed-fields) and [Runtime Triggers](/guides/query-builder/runtime-triggers).

## Framework Notes

::callout{type="success"}
**Next.js compatible.** The ORM uses `createRequire` internally to load `dbcube.config.js`, which works in both CJS and ESM contexts â€” including Next.js API routes and server components. Always call `dbcube.init()` on the server side only.
::

```typescript
// app/api/users/route.ts (Next.js App Router)
import { Dbcube } from 'dbcube';

const dbcube = new Dbcube();
dbcube.init();

export async function GET() {
  const users = await dbcube.database('myapp').table('users').get();
  return Response.json(users);
}
```

## ORM vs Query Builder: Which One?

| | `dbcube` (ORM) | `@dbcube/query-builder` |
|---|---|---|
| Config loading | Automatic from `dbcube.config.js` | Manual (you pass the database name) |
| Multiple databases | Built-in registry | One instance per database |
| Singleton | Yes | No |
| Best for | Applications | Libraries / minimal setups |

Both share the exact same query API â€” choosing one is purely about ergonomics.
