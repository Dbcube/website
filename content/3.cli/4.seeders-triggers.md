## seeder:add

Executes `.seeder.cube` files to insert data into your tables.

### Usage

```bash
# Run ALL seeders found under dbcube/
npx dbcube run seeder:add

# Run a single seeder by name
npx dbcube run seeder:add users
```

The filter matches the file name without the extension — `users` runs `users.seeder.cube` wherever it lives inside `dbcube/`. Passing `users.seeder` or `users.seeder.cube` also works.

### What It Does

1. Finds every `.seeder.cube` file under `dbcube/` (recursively).
2. Applies your name filter if you passed one.
3. Validates each file and its `@database` reference against `dbcube.config.js`.
4. Orders seeders respecting table dependencies (foreign-key targets are seeded first).
5. The schema-engine inserts the `@dataset` rows using the `@fields` column order.

### Example Output

```bash
$ npx dbcube run seeder:add

🌱 EXECUTING SEEDERS — myapp
✓ users
✓ categories
✓ products
─────────────────────────────
✅ 3 processed · 0 errors
```

::callout{type="info"}
**Seeders are inserts, not upserts.** Running the same seeder twice inserts the rows twice (unless a unique constraint rejects them). For a clean dataset, run `table:fresh` first.
::

### Typical Workflows

```bash
# Reset and reseed a development database
npx dbcube run table:fresh
npx dbcube run seeder:add

# Re-run just one dataset after editing it
npx dbcube run seeder:add categories
```

## trigger:fresh

Compiles and installs every `.trigger.cube` file.

### Usage

```bash
npx dbcube run trigger:fresh
```

### What It Does

1. Finds every `.trigger.cube` file under `dbcube/`.
2. Validates syntax and the `@database` reference.
3. Compiles each hook (`beforeAdd`, `afterAdd`, `beforeUpdate`, `afterUpdate`, `beforeDelete`, `afterDelete`) into an executable file under `dbcube/triggers/`:

```
dbcube/triggers/
├── myapp_users_beforeAdd.js
├── myapp_users_afterAdd.js
└── myapp_orders_afterUpdate.js
```

4. Registers them in the local config database so `useTriggers()` connections pick them up.

### Example Output

```bash
$ npx dbcube run trigger:fresh

⚡ EXECUTING TRIGGERS — myapp
✓ users
✓ orders
─────────────────────────────
✅ 2 processed · 0 errors
```

### Execution Logs

Every runtime execution is logged per database, table, and hook:

```
dbcube/logs/triggers/
└── myapp/
    ├── users_beforeAdd.log
    └── users_afterAdd.log
```

Tail these files when debugging trigger behavior:

```bash
# PowerShell
Get-Content dbcube/logs/triggers/myapp/users_beforeAdd.log -Tail 20 -Wait
```

::callout{type="warning"}
**Re-run after every trigger change.** Editing a `.trigger.cube` file has no effect until you run `trigger:fresh` again — the compiled files under `dbcube/triggers/` are what actually executes.
::
