## Installation

Install the Dbcube CLI globally to use it from anywhere in your system:

::code-group

```bash [npm]
npm install -g @dbcube/cli
```

```bash [yarn]
yarn global add @dbcube/cli
```

```bash [pnpm]
pnpm add -g @dbcube/cli
```

```bash [bun]
bun add -g @dbcube/cli
```


::

Verify the installation:

```bash
dbcube --version
```

## Basic Usage

The CLI follows a simple command structure:

```bash
dbcube [command] [subcommand] [options]
```

### Getting Help

Get help for any command:

```bash
# General help (lists every command with examples)
dbcube --help
dbcube -h
```

### Version Information

Check the CLI version:

```bash
dbcube --version
dbcube -v
```

## Available Commands

### Project Commands

```bash
# Scaffold a new project (config + dbcube/ + example cube + .gitignore)
dbcube init

# Generate TypeScript types from .table.cube files → dbcube/types.ts
dbcube generate

# Validate every .cube file without executing anything (exit 1 on errors — CI-friendly)
dbcube validate

# Diagnose setup: node, config, cubes, connectivity per database, daemons
dbcube doctor

# Watch mode: saving a .cube applies refresh/trigger automatically + regenerates types
dbcube dev
```

### Migration Commands

```bash
# Show applied and pending .alter.cube migrations
dbcube migrate:status

# Revert the last applied migration batch
dbcube migrate:rollback

# Generate .table.cube files FROM an existing database (introspection)
dbcube run pull [database]
```

Schema and data commands use the `dbcube run` prefix:

### Database Management

```bash
# Create database from configuration
dbcube run database:create
```

### Schema Operations

```bash
# DESTRUCTIVE: drop and recreate all tables (asks for written confirmation; --force to skip)
dbcube run table:fresh

# Non-destructive: create missing tables + apply diffs, data preserved
dbcube run table:refresh

# Apply pending .alter.cube migrations (--dry-run to preview SQL, --all to re-run)
dbcube run table:alter
```

### Seeder Operations

```bash
# Run all seeders
dbcube run seeder:add

# Run a single seeder by name (users.seeder.cube)
dbcube run seeder:add users
```

### Trigger Management

```bash
# Install/update all triggers
dbcube run trigger:fresh
```

### Binary Management

```bash
# Download database engine binaries (interactive)
dbcube run download

# Download a specific engine, optionally pinned to a version
dbcube run download query-engine
dbcube run download query-engine v3.1.1

# Check and update installed binaries
dbcube update
```

## Command Categories

### Project Commands

- `init` - Scaffold a new DBCube project
- `generate` - TypeScript types from `.cube` files
- `validate` - Validate all `.cube` files (CI)
- `doctor` - Diagnose configuration and connectivity
- `dev` - Watch mode for `.cube` files

### Database Commands

- `run database:create` - Create database from config file (interactive)
- `run database:create:config` - Add a database configuration only
- `run database:create:physical` - Create the physical database from an existing config
- `run pull [database]` - Introspect an existing database into `.table.cube` files

### Schema Commands

- `run table:fresh [--force] [--seeder]` - Drop and recreate all tables (written confirmation required)
- `run table:refresh` - Non-destructive: create missing tables + apply diffs
- `run table:alter [--dry-run] [--all]` - Apply pending `.alter.cube` migrations
- `migrate:status` / `migrate:rollback` - Migration history and rollback

### Data Commands

- `run seeder:add [name]` - Load all seeder files, or just one by name

### Trigger Commands

- `run trigger:fresh` - Install/update database triggers

### Binary Commands

- `run download [engine] [version]` - Download engine binaries (`query-engine`, `schema-engine`, `sqlite-engine`)
- `update` - Check versions and upgrade installed binaries

## Configuration

The CLI reads configuration from your `dbcube.config.js` file:

```javascript
// dbcube.config.js
require('dotenv').config({ quiet: true });

module.exports = function (config) {
  config.set({
    databases: {
      myapp: {
        type: "mysql",
        config: {
          HOST: process.env.DBCUBE_MYAPP_HOST,
          USER: process.env.DBCUBE_MYAPP_USER,
          PASSWORD: process.env.DBCUBE_MYAPP_PASSWORD,
          DATABASE: process.env.DBCUBE_MYAPP_DATABASE,
          PORT: parseInt(process.env.DBCUBE_MYAPP_PORT, 10)
        }
      }
    }
  });
};
```

::callout{type="info"}
You don't need to write this by hand — `dbcube run database:create` generates the entry and the matching `.env` variables for you. See the [Configuration guide](/getting-started/configuration) for the full format.
::

### Environment-Specific Configuration

Since the config file is plain JavaScript, branch on `NODE_ENV`:

```javascript
// dbcube.config.js
require('dotenv').config({ quiet: true });

module.exports = function (config) {
  const env = process.env.NODE_ENV || "development";

  const databases = env === "test"
    ? { main: { type: "sqlite", config: { DATABASE: "test" } } }
    : {
        main: {
          type: "mysql",
          config: {
            HOST: process.env.DBCUBE_MAIN_HOST,
            USER: process.env.DBCUBE_MAIN_USER,
            PASSWORD: process.env.DBCUBE_MAIN_PASSWORD,
            DATABASE: process.env.DBCUBE_MAIN_DATABASE,
            PORT: parseInt(process.env.DBCUBE_MAIN_PORT, 10)
          }
        }
      };

  config.set({ databases });
};
```

## File Structure

The CLI works with this file structure:

```
project/
├── dbcube.config.js          # Main configuration
└── dbcube/                   # Dbcube directory
    ├── users.table.cube      # Table schemas
    ├── posts.table.cube
    ├── users.seeder.cube     # Data seeders
    ├── categories.seeder.cube
    ├── users.trigger.cube    # Database triggers
    └── audit.trigger.cube
```

You can organize files in subdirectories if preferred:

```
project/
├── dbcube.config.js
└── dbcube/
    ├── tables/
    │   ├── users.table.cube
    │   └── posts.table.cube
    ├── seeders/
    │   └── users.seeder.cube
    └── triggers/
        └── users.trigger.cube
```

## Common Workflows

### Setting Up a New Project

```bash
# 1. Create dbcube.config.js manually or copy from template

# 2. Create database
dbcube run database:create

# 3. Create all tables
dbcube run table:fresh

# 4. Seed initial data
dbcube run seeder:add

# 5. Install triggers
dbcube run trigger:fresh
```

### Development Workflow

```bash
# 1. Make schema changes in .cube files

# 2. Update tables (safer - keeps data)
dbcube run table:refresh

# 3. Update triggers if needed
dbcube run trigger:fresh

# 4. Add test data
dbcube run seeder:add
```

### Production Deployment

```bash
# 1. Fresh install on production
dbcube run database:create
dbcube run table:fresh

# 2. Install production data
dbcube run seeder:add

# 3. Install triggers
dbcube run trigger:fresh
```

## Error Handling

The CLI provides helpful error messages with file and line context:

```bash
$ dbcube run table:fresh
❌ The cubes folder does not exist
# → create the dbcube/ directory with your .cube files

$ dbcube run table:fresh
[error] Database configuration 'myapp' not found in dbcube.config.js
# → the @database("myapp") in your cube file has no matching entry

$ dbcube badcommand
❌ Unknown command: badcommand
Run npx dbcube help to see all available commands.
```

## Troubleshooting

### Command Not Found

```bash
dbcube: command not found
```

**Solutions:**

- Reinstall CLI: `npm install -g @dbcube/cli`
- Check PATH: `echo $PATH`
- Use npx: `npx @dbcube/cli --help`

### Permission Errors

```bash
Error: EACCES: permission denied
```

**Solutions:**

- Fix npm permissions: `npm config set prefix ~/.npm-global`
- Use local installation: `npx dbcube run table:fresh`

### Configuration Errors

```bash
Error: Invalid configuration
```

**Solutions:**

- Validate `dbcube.config.js` syntax — it must export a function that calls `config.set()`
- Check the connection values in your `.env`
- Run `dbcube run database:create` — it recreates the config skeleton if missing
