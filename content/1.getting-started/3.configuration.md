## Configuration File

Dbcube looks for a `dbcube.config.js` file in your project root. This file exports a function that receives a configuration object and registers your databases with `config.set()`:

```javascript
// dbcube.config.js
require('dotenv').config({ quiet: true });

module.exports = function (config) {
  config.set({
    databases: {
      myapp: {
        type: "mysql",
        config: {
          HOST: process.env.DBCUBE_MYAPP_HOST,
          USER: process.env.DBCUBE_MYAPP_USER,
          PASSWORD: process.env.DBCUBE_MYAPP_PASSWORD,
          DATABASE: process.env.DBCUBE_MYAPP_DATABASE,
          PORT: parseInt(process.env.DBCUBE_MYAPP_PORT, 10)
        }
      }
    }
  });
};
```

The key of each entry inside `databases` (here `myapp`) is the **connection name** — the same name you pass to `new Database('myapp')`, `dbcube.database('myapp')`, and the one referenced by `@database("myapp")` in your `.cube` files.

::callout{type="info"}
**Don't want to write this by hand?** Run `npx dbcube run database:create` — the interactive wizard generates this exact structure and writes the matching variables to your `.env` file automatically.
::

## Configuration Parameters

Each database entry has two parts:

| Key | Type | Description |
|-----|------|-------------|
| `type` | string | Database engine: `mysql`, `postgres`, `sqlite`, or `mongodb` |
| `config` | object | Connection parameters (uppercase keys, see below) |

Connection parameters inside `config`:

| Parameter | Type | Required for | Description |
|-----------|------|--------------|-------------|
| `DATABASE` | string | All types | Database name (for SQLite: the file name, stored as `.dbcube/<name>.db`) |
| `HOST` | string | mysql, postgres, mongodb | Database server hostname |
| `PORT` | number | mysql, postgres, mongodb | Database server port |
| `USER` | string | mysql, postgres, mongodb | Database username |
| `PASSWORD` | string | mysql, postgres, mongodb | Database password |
| `URL` | string | — (optional) | Full connection string. When set it **overrides** HOST/USER/PASSWORD/PORT and enables **TLS** for managed/cloud databases. See [Cloud databases](#cloud-databases-url--tls). |
| `AUTH_TOKEN` | string | Turso only | Auth token for Turso / libSQL (used with a `libsql://` URL). |

::callout{type="tip"}
**Two ways to connect:** use the individual fields (`HOST`, `USER`, …) for local databases, or a single `URL` connection string for **managed/cloud** providers (Supabase, Neon, PlanetScale, MongoDB Atlas, Turso). The `URL` form turns on TLS automatically.
::

## Database Types

### MySQL

```javascript
config.set({
  databases: {
    myapp: {
      type: "mysql",
      config: {
        HOST: "localhost",
        PORT: 3306,
        USER: "root",
        PASSWORD: "password",
        DATABASE: "myapp_db"
      }
    }
  }
});
```

Default port: `3306`

### PostgreSQL

```javascript
config.set({
  databases: {
    analytics: {
      type: "postgres",
      config: {
        HOST: "localhost",
        PORT: 5432,
        USER: "postgres",
        PASSWORD: "password",
        DATABASE: "analytics_db"
      }
    }
  }
});
```

Default port: `5432`

### SQLite

```javascript
config.set({
  databases: {
    local: {
      type: "sqlite",
      config: {
        DATABASE: "local"   // Creates .dbcube/local.db
      }
    }
  }
});
```

SQLite databases are stored in the `.dbcube/` directory in your project root — only `DATABASE` is needed.

### MongoDB

```javascript
config.set({
  databases: {
    events: {
      type: "mongodb",
      config: {
        HOST: "localhost",
        PORT: 27017,
        USER: "admin",
        PASSWORD: "password",
        DATABASE: "events_db"
      }
    }
  }
});
```

Default port: `27017`

## Cloud databases (URL + TLS)

DBCube connects to **managed/cloud databases over TLS** using a single `URL`
connection string. When `URL` is present it overrides the individual
`HOST`/`USER`/`PASSWORD`/`PORT` fields and negotiates an encrypted connection
automatically — no extra flags needed.

::callout{type="success"}
Supabase, Neon, AWS RDS, PlanetScale, Aiven, MongoDB Atlas and **Turso** all work
out of the box. The same DBCube API and `.cube` schemas run against local *and*
cloud databases — only the config entry changes.
::

### PostgreSQL — Supabase / Neon / RDS

```javascript
config.set({
  databases: {
    app: {
      type: "postgres",
      config: {
        URL: process.env.DATABASE_URL,
        // e.g. postgresql://postgres:****@db.xxxx.supabase.co:5432/postgres?sslmode=require
        DATABASE: "postgres"
      }
    }
  }
});
```

Add `?sslmode=require` to enforce TLS (recommended for any public endpoint).

### MySQL — PlanetScale / Aiven

```javascript
config.set({
  databases: {
    app: {
      type: "mysql",
      config: {
        URL: process.env.DATABASE_URL
        // e.g. mysql://user:****@aws.connect.psdb.cloud/mydb?ssl-mode=REQUIRED
      }
    }
  }
});
```

### MongoDB — Atlas

```javascript
config.set({
  databases: {
    app: {
      type: "mongodb",
      config: {
        URL: process.env.MONGODB_URI,
        // e.g. mongodb+srv://user:****@cluster.mongodb.net/?retryWrites=true
        DATABASE: "myapp"   // select the database to use
      }
    }
  }
});
```

::callout{type="info"}
For MongoDB, set `DATABASE` alongside `URL` so DBCube knows which database to use,
even if the connection string already names one.
::

### Turso / libSQL

DBCube speaks the **Turso (libSQL)** remote protocol. Use `type: "sqlite"` with a
`libsql://` URL and your auth token:

```javascript
config.set({
  databases: {
    edge: {
      type: "sqlite",
      config: {
        URL: process.env.TURSO_DATABASE_URL,   // libsql://your-db.turso.io
        AUTH_TOKEN: process.env.TURSO_AUTH_TOKEN,
        DATABASE: "edge"
      }
    }
  }
});
```

The full SQLite feature set (reads, writes, transactions, raw SQL) works against
Turso exactly as it does with a local SQLite file — DBCube routes to the remote
libSQL client automatically when the URL starts with `libsql://`.

### `.env` for cloud

```bash
# .env
DATABASE_URL=postgresql://postgres:****@db.xxxx.supabase.co:5432/postgres?sslmode=require
MONGODB_URI=mongodb+srv://user:****@cluster.mongodb.net/?retryWrites=true
TURSO_DATABASE_URL=libsql://your-db.turso.io
TURSO_AUTH_TOKEN=eyJhbGciOi...
```

::callout{type="warning"}
A connection string contains your password. Always load it from `process.env`
(never hard-code it) and keep `.env` out of version control.
::

## Multiple Databases

All your databases live in the same `databases` object:

```javascript
// dbcube.config.js
require('dotenv').config({ quiet: true });

module.exports = function (config) {
  config.set({
    databases: {
      primary: {
        type: "mysql",
        config: {
          HOST: process.env.DBCUBE_PRIMARY_HOST,
          USER: process.env.DBCUBE_PRIMARY_USER,
          PASSWORD: process.env.DBCUBE_PRIMARY_PASSWORD,
          DATABASE: process.env.DBCUBE_PRIMARY_DATABASE,
          PORT: parseInt(process.env.DBCUBE_PRIMARY_PORT, 10)
        }
      },
      analytics: {
        type: "postgres",
        config: {
          HOST: process.env.DBCUBE_ANALYTICS_HOST,
          USER: process.env.DBCUBE_ANALYTICS_USER,
          PASSWORD: process.env.DBCUBE_ANALYTICS_PASSWORD,
          DATABASE: process.env.DBCUBE_ANALYTICS_DATABASE,
          PORT: parseInt(process.env.DBCUBE_ANALYTICS_PORT, 10)
        }
      },
      cache: {
        type: "sqlite",
        config: {
          DATABASE: process.env.DBCUBE_CACHE_DATABASE
        }
      }
    }
  });
};
```

Then use them in your code:

```typescript
import Database from '@dbcube/query-builder';

const primaryDb = new Database("primary");
const users = await primaryDb.table('users').get();

const analyticsDb = new Database("analytics");
const stats = await analyticsDb.table('stats').get();
```

Or with the ORM:

```typescript
import { Dbcube } from 'dbcube';

const dbcube = new Dbcube();
dbcube.init();

const users = await dbcube.database('primary').table('users').get();
```

## Environment Variables

The CLI wizard follows the naming convention `DBCUBE_<CONNECTION-NAME>_<PARAM>`. A generated `.env` looks like this:

```bash
# .env
DBCUBE_MYAPP_HOST=localhost
DBCUBE_MYAPP_PORT=3306
DBCUBE_MYAPP_DATABASE=myapp_db
DBCUBE_MYAPP_USER=root
DBCUBE_MYAPP_PASSWORD=password

# SQLite connections only need the database name
DBCUBE_CACHE_DATABASE=cache
```

You are free to use your own variable names — the convention just keeps multiple connections tidy.

### Environment-Specific Configuration

Since `dbcube.config.js` is plain JavaScript, you can branch on `NODE_ENV`:

```javascript
require('dotenv').config({ quiet: true });

module.exports = function (config) {
  const env = process.env.NODE_ENV || 'development';

  const databases = env === 'test'
    ? {
        main: {
          type: "sqlite",
          config: { DATABASE: "test" }
        }
      }
    : {
        main: {
          type: "mysql",
          config: {
            HOST: process.env.DBCUBE_MAIN_HOST,
            USER: process.env.DBCUBE_MAIN_USER,
            PASSWORD: process.env.DBCUBE_MAIN_PASSWORD,
            DATABASE: process.env.DBCUBE_MAIN_DATABASE,
            PORT: parseInt(process.env.DBCUBE_MAIN_PORT, 10)
          }
        }
      };

  config.set({ databases });
};
```

Load a different `.env` per environment if you prefer:

```javascript
const env = process.env.NODE_ENV || 'development';
require('dotenv').config({ path: `.env.${env}` });
```

## Common Mistakes

::callout{type="warning"}
**The config function must be the module export and call `config.set()`.** If `dbcube.config.js` doesn't export a function, you'll see: `❌ The dbcube.config.js file does not export a function.`
::

```javascript
// ✗ Wrong — exporting an object
module.exports = { databases: { ... } };

// ✗ Wrong — lowercase keys are not read
config.set({ databases: { myapp: { type: "mysql", config: { host: "..." } } } });

// ✓ Correct — function export + uppercase connection keys
module.exports = function (config) {
  config.set({ databases: { myapp: { type: "mysql", config: { HOST: "..." } } } });
};
```

Other things to check when a connection fails:

- The connection name in `@database("...")` (cube files) and `database('...')` (code) must match a key under `databases` exactly.
- `PORT` should be a number — wrap env vars with `parseInt(..., 10)`.
- The config file must be in the **working directory** where your app or the CLI runs (`process.cwd()`).

## Troubleshooting

**Connection refused** (`ECONNREFUSED`)

Verify the database server is running and check `HOST`/`PORT`:

```bash
# Check if MySQL is running
sudo systemctl status mysql

# Test connection manually
mysql -h localhost -u root -p
```

**Authentication failed** (`Access denied`)

Verify `USER` and `PASSWORD` are correct and the user has privileges on the target database.

**Database configuration not found**

```
Database configuration 'myapp' not found in dbcube.config.js
```

The name referenced (in code or in a `.cube` file's `@database`) doesn't exist as a key under `databases`. Check for typos and case differences.

**Config file not found**

```
❌ Config file not found, please create a dbcube.config.js file
```

Create `dbcube.config.js` in the project root, or run `npx dbcube run database:create` to generate it.

**SQLite file location**

SQLite files live in `.dbcube/<DATABASE>.db` and the directory is created automatically. Ensure the process has write permissions in the project root.

## Security Best Practices

**Never commit credentials** — add to `.gitignore`:

```bash
.env
.env.*
.dbcube/
```

**Use environment variables:**

```javascript
// ✓ Good
PASSWORD: process.env.DBCUBE_MYAPP_PASSWORD

// ✗ Bad
PASSWORD: "my_password_123"
```

**Use different credentials per environment** and rotate passwords regularly.
