## What's here

Two kinds of examples, all in **TypeScript**:

1. **Feature tour** — bite-sized pages, one per capability, mapped to the
   runnable files in the public
   [**github.com/Dbcube/examples**](https://github.com/Dbcube/examples) repo. They
   use the **`dbcube` all-in-one package** and a single SQLite database (`demo`),
   so you need **zero external services** to follow along.
2. **Real-world use cases** — complete, copy-paste features (a blog API, a
   checkout, auth, a dashboard, multi-database) showing how the pieces fit
   together in code you'd actually ship.

## Setup

Clone the repo, install, and run any example:

```bash
git clone https://github.com/Dbcube/examples
cd examples
npm install
node setup.js              # creates a fresh demo.db (SQLite) with sample data
npx tsx 01-quickstart.ts   # run any example (tsx runs TypeScript directly)
```

The snippets in this section are TypeScript. Run them with
[`tsx`](https://github.com/privatenumber/tsx) (`npx tsx file.ts`) or compile with
`tsc` — Dbcube ships its own types, no `@types` package needed.

The setup builds three tables (`users`, `orders`, `settings`) with sample rows.
In a real project you would define these in `.cube` files and run
`npx dbcube run table:refresh` — see the [Schema guides](/guides/schemas/overview).

## Configuration

Every example reads `dbcube.config.js`. The simple examples use a single SQLite
entry; the full test suite adds real MySQL / PostgreSQL / MongoDB entries backed
by `docker-compose.yml`:

```ts [dbcube.config.js]
module.exports = function (config) {
  config.set({
    databases: {
      // Simple examples — no services needed
      demo: {
        type: "sqlite",
        config: { DATABASE: "demo" }
      },

      // Real engines (docker compose up -d --wait)
      ft_postgres: {
        type: "postgres",
        config: {
          HOST: "127.0.0.1", USER: "postgres", PASSWORD: "dbcube_test",
          DATABASE: "fulltest", PORT: 30046
        },
        pool: { maxConnections: 8, acquireTimeoutMs: 5000 },
        daemon: { requestTimeoutMs: 60000 }
      }
      // ft_mysql, ft_mongo … see the file
    }
  });
};
```

You get a connection with `dbcube.database('<name>')`, where `<name>` is a key
from the `databases` map. The **same code** runs against any engine — only the
config entry changes.

## The full test suite

Beyond the numbered examples, two files exercise *everything* against real
engines:

- **`full-test.js`** — 65 assertions per engine covering reads, writes, edge
  cases (nulls, unicode, 100 KB strings, numeric limits), aggregations,
  pagination, transactions and bulk inserts. Run `node full-test.js postgres`
  (or `mysql` / `sqlite` / `mongo`).
- **`run-all.js`** — runs `full-test.js` against every configured engine plus the
  CLI test suite, and prints a global summary.

```bash
docker compose up -d --wait     # start MySQL + PostgreSQL + MongoDB
node run-all.js                 # full matrix across all engines
```

::callout{type="info"}
These suites are the same ones Dbcube runs in CI before every release — they are
the guarantee that the feature set documented here actually works on all four
engines.
::
