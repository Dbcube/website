`.alter.cube` files are Dbcube's answer to surgical migrations. While `table:fresh` recreates tables and `table:refresh` diffs your `.table.cube` against the last applied schema, **alter files let you state the exact change you want** — rename a column, change a type, tweak options — and apply it with one command, preserving all data.

<!-- DIAGRAM: migration-strategies (see diagrams.md) -->

## When to Use Which

| Strategy | Command | Data preserved | Best for |
|----------|---------|----------------|----------|
| Fresh | `dbcube run table:fresh` | ✗ No | New environments, development resets |
| Refresh | `dbcube run table:refresh` | ✓ Yes | Additive changes detected automatically |
| **Alter** | `dbcube run table:alter` | ✓ Yes | **Explicit changes: renames, type changes, drops** |

## File Structure

An alter file targets one table and lists one or more directives:

```ts
@database("myapp");
@table("users");

@addColumn({
    phone: {
        type: "varchar";
        length: "20";
        options: ["null"];
    };
});

@renameColumn({last: "last_name"; new: "surname"});
```

Save it as `<anything>.alter.cube` inside your `dbcube/` directory, then run:

```bash
npx dbcube run table:alter
```

## Directive Reference

### `@changeName` — Rename the Table

```ts
@changeName({last: "users"; new: "customers"});
```

### `@addColumn` — Add a Column

Uses the exact same column syntax as `.table.cube`:

```ts
@addColumn({
    avatar_url: {
        type: "varchar";
        length: "500";
        options: ["null"];
    };
    login_count: {
        type: "int";
        defaultValue: "0";
        options: ["not null"];
    };
});
```

### `@deleteColumn` — Drop a Column

```ts
@deleteColumn("legacy_field");
```

::callout{type="warning"}
Dropping a column permanently deletes its data. Back up first if there is any chance you'll need it.
::

### `@renameColumn` — Rename a Column

```ts
@renameColumn({last: "created"; new: "created_at"});
```

### `@changeType` — Change a Column's Type

```ts
@changeType({
    description: {last: "varchar"; new: "text"};
});
```

### `@changeLength` — Change a Column's Length

```ts
@changeLength({
    title: {last: "255"; new: "500"};
});
```

### `@changeDefault` — Change the Default Value

```ts
@changeDefault({
    status: {last: "pending"; new: "active"};
});
```

### `@changeOptions` — Add or Remove Options

```ts
@changeOptions({
    email: {
        add: ["unique"];
        remove: ["null"];
    };
});
```

### `@changeEnumValues` — Replace Enum Values

```ts
@changeEnumValues({
    status: {
        values: ["draft", "published", "archived", "deleted"];
    };
});
```

## Complete Example

A realistic migration evolving a `products` table:

```ts
@database("shop");
@table("products");

// 1. New column for SEO
@addColumn({
    slug: {
        type: "varchar";
        length: "255";
        options: ["unique", "null"];
    };
});

// 2. Descriptions outgrew varchar
@changeType({
    description: {last: "varchar"; new: "text"};
});

// 3. Clearer naming
@renameColumn({last: "qty"; new: "stock_quantity"});

// 4. New lifecycle state
@changeEnumValues({
    status: {
        values: ["draft", "active", "discontinued"];
    };
});

// 5. Remove deprecated column
@deleteColumn("internal_notes");
```

```bash
$ npx dbcube run table:alter

🔧 EXECUTING ALTER TABLES — shop
✓ products
─────────────────────────────
✅ 1 processed · 0 errors
```

## How It Works

1. The schema-builder finds every `.alter.cube` file under `dbcube/` (subdirectories included).
2. Each file is validated — syntax plus the `@database` reference against `dbcube.config.js`.
3. The **schema-engine** (Rust) parses the directives (`parse_alter`), generates dialect-specific statements (`generate --mode alter`), and executes them (`execute --mode alter`).
4. Per-engine translation is automatic:
   - **MySQL** → `ALTER TABLE ... MODIFY/CHANGE/ADD/DROP`
   - **PostgreSQL** → separate `ALTER COLUMN` statements per change
   - **SQLite** → table-rebuild strategy (create temp table → copy data → swap) for changes SQLite can't do natively
   - **MongoDB** → `$set`/`$unset`/`$rename` update pipelines

## Migration History, Dry-Run & Rollback

`table:alter` tracks every applied file in `dbcube/migrations.json` (commit it — the history travels with your repo):

```bash
# Preview the SQL without touching the database
npx dbcube run table:alter --dry-run

# Apply only PENDING migrations (already-applied files are skipped)
npx dbcube run table:alter

# Inspect the state
npx dbcube migrate:status
#   Applied:
#     ✓ add-product-slug.alter.cube   batch 1 · 2026-06-11 · reversible
#   Pending:
#     ○ rename-qty.alter.cube

# Revert the last batch (auto-generated reverse directives)
npx dbcube migrate:rollback
```

Rollback inverts each directive automatically — `@addColumn` becomes `@deleteColumn`, every `{last/new}` pair swaps, `@changeOptions` swaps add/remove. `@deleteColumn` and `@changeEnumValues` are irreversible: rollback refuses with a clear message instead of guessing.

`migrate:status` also warns when an **applied file was edited afterwards** (hash mismatch) — write a new alter file instead of editing applied ones.

## Workflow Recommendations

1. **One concern per file** — `2026-06-add-product-slug.alter.cube` is easier to review and reason about than a mega-file.
2. **Date-prefix your files** so the history reads chronologically in your repo.
3. **Update the matching `.table.cube` afterwards** so fresh environments build the final structure directly. The alter file documents *how* you got there; the table file documents *where you are*.
4. **Commit `dbcube/migrations.json`** — teammates' `table:alter` will then skip what's already applied and run only what's new.
