Triggers let you attach business logic to write operations. You declare them once in `.trigger.cube` files, install them with the CLI, and the query builder executes them automatically around every `insert()`, `update()`, and `delete()` — in your Node.js process, for **any** of the four database engines (including MongoDB and SQLite, which have limited or no native trigger support).

<!-- DIAGRAM: trigger-lifecycle (see diagrams.md) -->

## The Lifecycle

| Hook | Fires | Typical use |
|------|-------|-------------|
| `beforeAdd` | Before each INSERT | Validation, hashing passwords, auto-filling fields |
| `afterAdd` | After each INSERT | Notifications, audit logs, related records |
| `beforeUpdate` | Before each UPDATE | Validation, conflict checks, `updated_at` stamps |
| `afterUpdate` | After each UPDATE | Audit trails, cache invalidation |
| `beforeDelete` | Before each DELETE | Dependency checks, blocking protected rows |
| `afterDelete` | After each DELETE | Cleanup, cascade operations, logging |

## 1. Declare Triggers in `.trigger.cube`

```cube
@database("myapp");
@table("users");

@beforeAdd({
    name: "validate_email";
    description: "Reject invalid emails before they reach the database";
    function: ({db, oldData, newData, gdb}) => {
        const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
        if (!emailRegex.test(newData.email)) {
            throw new Error('Invalid email format');
        }
        newData.created_at = new Date();
    };
});

@afterAdd({
    name: "welcome_log";
    description: "Log every new registration";
    function: ({db, oldData, newData, gdb}) => {
        console.log(`New user: ${newData.email}`);
    };
});
```

Each trigger function receives:

| Parameter | Description |
|-----------|-------------|
| `db` | Connection to the same database, for follow-up queries |
| `oldData` | Record state before the operation (`null` on insert) |
| `newData` | Incoming record state (`null` on delete) |
| `gdb` | Global database access (query other configured databases) |

## 2. Install Them

```bash
npx dbcube run trigger:fresh
```

This compiles every trigger into `dbcube/triggers/<database>_<table>_<hook>.js` and registers it in the local config database. Execution logs are written to `dbcube/logs/triggers/<database>/`.

## 3. Enable Them at Runtime

Triggers are opt-in per connection via `useTriggers()`:

```typescript
import { Dbcube } from 'dbcube';

const dbcube = new Dbcube();
dbcube.init();

const db = await dbcube.database('myapp').useTriggers();

// beforeAdd → INSERT → afterAdd, per record
await db.table('users').insert([
  { name: 'John', email: 'john@example.com' }
]);
```

::callout{type="info"}
**Per-record execution.** When you insert an array of records with triggers enabled, Dbcube processes them one by one so every record passes through the full hook chain individually.
::

## Commit / Discard Semantics

`before*` hooks act as **interceptors**:

1. The `before` hook runs first. If it **throws**, the database operation is never executed.
2. The database operation runs.
3. If the operation **fails**, the interceptor is **discarded** — side effects tracked by the trigger system are rolled back and the error is reported with code context.
4. If the operation succeeds, the interceptor **commits**.

`after*` hooks run only when the database operation succeeded.

```typescript
try {
  await db.table('users').insert([{ name: 'X', email: 'not-an-email' }]);
} catch (error) {
  // beforeAdd threw → nothing was written to the database
}
```

## Triggers + Query Builder = Portable Logic

Because triggers run in Node.js rather than inside the database server:

- The **same trigger code** works on MySQL, PostgreSQL, SQLite, and MongoDB.
- You can use any npm package inside trigger functions (email, queues, validation libraries).
- Triggers are version-controlled with your code, reviewed in PRs, and deployed atomically.

::callout{type="warning"}
**They run in-process.** Triggers only fire for writes that go through the Dbcube query builder. Writes made directly via another client (e.g. raw SQL in a DB GUI) bypass them — same trade-off as any application-level hook system.
::

## Combining with Computed Fields

```typescript
const db = await dbcube.database('myapp').useComputes();
const dbT = await dbcube.database('myapp').useTriggers();
```

`useComputes()` and `useTriggers()` each return a configured connection. Use the one (or both, sequentially) that the code path needs — read-heavy paths can skip trigger overhead entirely.

## Debugging Triggers

- **Logs**: every execution appends to `dbcube/logs/triggers/<database>/<table>_<hook>.log`.
- **Compiled code**: inspect `dbcube/triggers/*.js` to see exactly what will run.
- **Errors**: failures surface in your application with the `[error]` / `[code]` formatted output, pointing at the offending line.
