## What are .cube Files?

`.cube` files are specialized schema definition files that use an annotation-based syntax. They are more readable than SQL migrations and provide better tooling support than traditional ORM configurations.

### Advantages of .cube Files

- **Readability**: Clear and expressive syntax
- **IntelliSense**: Auto-completion and validation in VS Code
- **Relationships**: Simple foreign key and relationship definitions
- **Productivity**: Built-in snippets and templates
- **Validation**: Real-time syntax verification
- **Focused**: Each file has a specific purpose

## Types of .cube Files

Dbcube supports three main types of `.cube` files:

### 1. `.table.cube` - Table Definitions

Define table structure, columns, data types, and relationships.

::cube-code
---
filename: schema/users.cube
code: |
  @database("myapp");

  @meta({
    name: "users";
    description: "User accounts and profiles";
  });

  @columns({
    id: {
      type: "int";
      options: ["primary", "autoincrement"];
    };
    name: {
      type: "varchar";
      length: "255";
      options: ["not null"];
    };
    email: {
      type: "varchar";
      length: "255";
      options: ["not null", "unique"];
    };
  });
---
::

### 2. `.seeder.cube` - Initial Data

Configure test data and initial database content.

::cube-code
---
filename: schema/users.seeder.cube
code: |
  @database("myapp");
  @table("users");

  @fields("name", "email", "status");

  @dataset(
    ("John Doe", "john@example.com", "active"),
    ("Jane Smith", "jane@example.com", "active"),
    ("Bob Wilson", "bob@example.com", "inactive")
  );
---
::

### 3. `.trigger.cube` - Triggers and Logic

Define database event handlers and business logic.

::cube-code
---
filename: schema/users.trigger.cube
code: |
  @database("myapp");
  @table("users");

  @beforeAdd({
    name: "validate_email";
    description: "Validate email format before insertion";
    function: ({db, oldData, newData, gdb}) => {
      const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
      if (!emailRegex.test(newData.email)) {
        throw new Error('Invalid email format');
      }
    };
  });
---
::

## Project Structure

Organize your `.cube` files in a logical structure:

```
project/
├── dbcube/
│   ├── users.table.cube
│   ├── roles.table.cube
│   ├── posts.table.cube
│   ├── users.seeder.cube
│   ├── posts.seeder.cube
│   └── users.trigger.cube
└── dbcube.config.js
```

You can organize files in subdirectories if needed:

```
project/
├── dbcube/
│   ├── tables/
│   │   ├── users.table.cube
│   │   ├── roles.table.cube
│   │   └── posts.table.cube
│   ├── seeders/
│   │   ├── users.seeder.cube
│   │   └── posts.seeder.cube
│   └── triggers/
│       └── users.trigger.cube
└── dbcube.config.js
```

## Basic Concepts

### Annotations

Annotations define metadata and configurations:

::cube-code
---
filename: schema/example.cube
code: |
  @database("db_name");      // Target database
  @table("table_name");      // Target table (for seeders/triggers)
  @meta({ ... });           // Table metadata
  @columns({ ... });        // Column definitions
  @fields(...);             // Fields for seeders
  @dataset(...);            // Data for seeders
---
::

### Metadata

Provides descriptive information about tables:

::cube-code
---
filename: schema/posts.cube
code: |
  @meta({
    name: "posts";
    description: "Blog posts with markdown content";
    version: "1.0";
    author: "Development Team";
  });
---
::

### Automatic Validation

Dbcube automatically validates:

- ✅ Annotation syntax
- ✅ Valid data types
- ✅ Compatible column options
- ✅ Foreign key references
- ✅ Required lengths for VARCHAR
- ✅ Valid enum values

## VS Code Integration

The Dbcube VS Code extension (`cube-lang`) provides:

### Syntax Highlighting

- Keywords and annotations are highlighted
- Data types have distinct colors
- Strings and values are properly formatted

### Auto-completion

- IntelliSense for annotations
- Data type suggestions
- Column option completions
- Contextual suggestions

### Real-time Validation

- Syntax error detection
- Type checking for column definitions
- Database configuration validation
- Required field validation

### Commands

Available through the Command Palette (Ctrl+Shift+P):

- **Validate Cube File** - Check syntax and structure
- **Format Cube Document** - Auto-format .cube files

## Workflow

### Development Workflow

```bash
# 1. Create/modify .cube files in dbcube/ directory

# 2. Create tables (drops and recreates)
dbcube run table:fresh

# 3. Or refresh tables (updates without dropping)
dbcube run table:refresh

# 4. Seed data
dbcube run seeder:add

# 5. Install triggers
dbcube run trigger:fresh
```

### Common Commands

```bash
# Create database
dbcube run database:create

# Recreate all tables from scratch
dbcube run table:fresh

# Update table structures (safer)
dbcube run table:refresh

# Load seed data
dbcube run seeder:add

# Install/update triggers
dbcube run trigger:fresh
```

## Advantages Over Other Approaches

### vs. Traditional SQL Migrations

- ✅ More readable and maintainable
- ✅ Automatic syntax validation
- ✅ IDE auto-completion
- ✅ Clearer versioning

### vs. ORM Configurations

- ✅ Simpler syntax
- ✅ Less verbose
- ✅ Declarative approach
- ✅ Better separation of concerns

### vs. Graphical Tools

- ✅ Native version control
- ✅ Team collaboration
- ✅ Automation and CI/CD
- ✅ Exact reproducibility
