`with()` attaches related records to your results. Dbcube reads the `foreign` definitions in your `.table.cube` files to figure out the relationship automatically, and loads each relation with **one batched `whereIn` query** — never one query per row (the classic N+1 problem).

## Automatic Relations from .cube Files

Given these schemas:

```ts
// users.table.cube
@database("myapp");
@meta({ name: "users"; description: "Users"; });
@columns({
    id: { type: "int"; options: ["primary", "autoincrement"]; };
    name: { type: "varchar"; length: "255"; options: ["not null"]; };
});

// orders.table.cube
@database("myapp");
@meta({ name: "orders"; description: "Orders"; });
@columns({
    id: { type: "int"; options: ["primary", "autoincrement"]; };
    user_id: {
        type: "int";
        options: ["not null", "index"];
        foreign: { table: "users"; column: "id"; };
    };
    total: { type: "decimal"; length: "10,2"; options: ["not null"]; };
});
```

### hasMany — parent with children

`orders.user_id` points at `users.id`, so from `users` the relation is **one-to-many**:

```typescript
const users = await db.table('users').with('orders').get();

// [
//   { id: 1, name: 'Ana',  orders: [{ id: 10, user_id: 1, total: 99.9 }, ...] },
//   { id: 2, name: 'Luis', orders: [] }
// ]
```

### belongsTo — child with its parent

From `orders`, the same foreign key resolves as **many-to-one**, attaching a single object:

```typescript
const orders = await db.table('orders')
  .with('users', { type: 'one' })
  .get();

// [{ id: 10, total: 99.9, users: { id: 1, name: 'Ana' } }, ...]
```

## Explicit Options

When the relation name doesn't match the table, or there's no `foreign` metadata, configure it explicitly:

```typescript
const posts = await db.table('posts')
  .with('author', {
    table: 'users',        // related table
    foreignKey: 'author_id', // FK column
    localKey: 'id',          // key it points to (default 'id')
    type: 'one',             // 'one' (belongsTo) | 'many' (hasMany)
  })
  .get();

// [{ id: 1, title: '...', author: { id: 7, name: 'Ana' } }]
```

| Option | Default | Description |
|--------|---------|-------------|
| `table` | the relation name | Related table |
| `foreignKey` | from `.cube` metadata, or `<singular>_id` | FK column |
| `localKey` | `'id'` | Referenced key |
| `type` | auto-detected | `'many'` → array, `'one'` → object or `null` |

## Multiple Relations

Chain `with()` as many times as needed — each relation costs exactly one extra query:

```typescript
const users = await db.table('users')
  .with('orders')
  .with('posts')
  .where('status', '=', 'active')
  .limit(20)
  .get();
```

## How It Works (No N+1)

```typescript
const users = await db.table('users').with('orders').get();
```

1. Main query runs: `SELECT * FROM users` → 20 rows.
2. Dbcube collects the ids: `[1, 2, ..., 20]`.
3. **One** batched query: `SELECT * FROM orders WHERE user_id IN (1, 2, ..., 20)`.
4. Results are grouped in memory and attached to each user.

2 queries total — with a per-row approach it would have been 21.

## Notes

- `with()` applies to `get()`. For a single record, use `where(...).with(...).get()` and take the first element.
- Rows without matches get `[]` (many) or `null` (one) — never `undefined`.
- Relation queries run on the same connection (and inside the same transaction when using `trx`).
- The `.cube` metadata scan is cached per process.
