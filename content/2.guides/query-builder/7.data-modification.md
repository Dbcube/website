## Data Insertion

### insert()

`insert()` **always receives an array** of objects, even for a single record:

```typescript
// Single record — still wrapped in an array
await db.table('users').insert([
  { name: 'Alice', email: 'alice@example.com', age: 28 }
]);

// Multiple records in one call
await db.table('users').insert([
  { name: 'Bob', email: 'bob@example.com', age: 32 },
  { name: 'Carol', email: 'carol@example.com', age: 27 },
]);
```

Passing anything that isn't an array of plain objects throws:

```
Error: The insert method requires an array of objects with key-value pairs.
Error: The array must contain only valid objects.
```

The return value is the inserted data:

```typescript
const inserted = await db.table('users').insert([
  { name: 'Dave', email: 'dave@example.com' }
]);
```

### Defaults and Auto Columns

Columns with `defaultValue`, `autoincrement`, or computed values in your `.cube` schema can be omitted from the payload:

```typescript
// id (autoincrement) and status (defaultValue: "active") not needed
await db.table('users').insert([
  { name: 'Eve', email: 'eve@example.com' }
]);
```

## Data Update

### update()

`update()` receives a **plain object** with the fields to change, and **requires at least one WHERE condition**:

```typescript
await db.table('users')
  .where('id', '=', 1)
  .update({ name: 'John Updated', age: 31 });
```

```typescript
// ✗ Throws — no condition
await db.table('users').update({ status: 'inactive' });
// Error: You must specify at least one WHERE condition to perform an update.

// ✗ Throws — array is not valid for update
await db.table('users').where('id', '=', 1).update([{ name: 'X' }]);
// Error: The update method requires an object with key-value pairs.
```

### Updating Multiple Rows

Any rows matching the conditions are updated:

```typescript
// Deactivate every user inactive for a year
await db.table('users')
  .where('last_login', '<', '2025-06-10')
  .where('status', '=', 'active')
  .update({ status: 'dormant' });
```

### Conditional Update Patterns

```typescript
// Update by membership
await db.table('orders')
  .whereIn('id', [10, 11, 12])
  .update({ status: 'shipped' });

// Update with a group condition
await db.table('products')
  .whereGroup(q => {
    q.where('stock', '=', 0).orWhere('discontinued', '=', true);
  })
  .update({ visible: false });
```

## Upsert (Insert or Update)

`upsert()` inserts rows, but **updates them instead when a conflict occurs** on the given keys — one statement, fully atomic:

```typescript
// 'key' has a UNIQUE constraint
await db.table('settings').upsert(
  [
    { key: 'theme', value: 'dark' },
    { key: 'lang', value: 'es' },
  ],
  ['key']                  // conflict key(s)
);

// Only overwrite specific columns on conflict
await db.table('products').upsert(
  rows,
  ['sku'],
  ['price', 'stock']       // name/description stay untouched on update
);
```

| Engine | Generated SQL |
|--------|---------------|
| MySQL | `INSERT ... ON DUPLICATE KEY UPDATE` |
| PostgreSQL / SQLite | `INSERT ... ON CONFLICT (...) DO UPDATE SET` |
| MongoDB | Not supported yet (clear error) |

::callout{type="warning"}
The conflict keys must have a **UNIQUE or PRIMARY KEY constraint** in the table — that's what the database uses to detect the conflict.
::

## Atomic Counters

`increment()` / `decrement()` generate `SET col = col ± n` — the database does the math, so concurrent requests can't race each other:

```typescript
// Stock control without read-modify-write races
await db.table('products').where('id', '=', 5).decrement('stock', 1);

// Views counter +1, updating a timestamp in the same statement
await db.table('posts')
  .where('id', '=', postId)
  .increment('views', 1, { last_viewed_at: new Date().toISOString() });
```

Both require at least one WHERE condition, like `update()`.

```typescript
// ✗ The racy pattern these methods replace:
const p = await db.table('products').find(5);
await db.table('products').where('id', '=', 5).update({ stock: p.stock - 1 });
// Two concurrent requests both read stock=10 → both write 9. One sale lost.
```

## Data Deletion

### delete()

Like `update()`, `delete()` **requires at least one WHERE condition**:

```typescript
await db.table('users')
  .where('id', '=', 1)
  .delete();
```

```typescript
// ✗ Throws — full-table deletes are blocked
await db.table('users').delete();
// Error: You must specify at least one WHERE condition to perform a delete.
```

To intentionally clear a table, use `truncate()` — the only write allowed without WHERE, because the destructive intent is explicit in the name:

```typescript
await db.table('temp_imports').truncate();   // deletes ALL rows
```

### Soft Deletes (Pattern)

Prefer marking rows over destroying them, using `update()`:

```typescript
// "Delete"
await db.table('posts')
  .where('id', '=', postId)
  .update({ deleted_at: new Date().toISOString() });

// Queries then exclude soft-deleted rows
const visible = await db.table('posts').whereNull('deleted_at').get();
```

## Write Operations and Triggers

When the connection is created with `useTriggers()`, every write passes through the trigger lifecycle **per record**:

```typescript
const db = await new Database('myapp').useTriggers();

await db.table('users').insert([
  { name: 'Frank', email: 'frank@example.com' }
]);
// beforeAdd → INSERT → afterAdd

await db.table('users').where('id', '=', 5).update({ status: 'banned' });
// beforeUpdate → UPDATE → afterUpdate

await db.table('users').where('id', '=', 5).delete();
// beforeDelete → DELETE → afterDelete
```

A `before*` trigger that throws **cancels the operation** — nothing reaches the database. See [Runtime Triggers](/guides/query-builder/runtime-triggers).

## Validation Patterns

The builder validates shape (arrays/objects, mandatory WHERE), but business validation belongs in your code or in triggers:

### In Application Code

```typescript
function assertValidUser(user: { email?: string; age?: number }) {
  if (!user.email?.includes('@')) throw new Error('Invalid email');
  if (user.age !== undefined && user.age < 0) throw new Error('Invalid age');
}

assertValidUser(payload);
await db.table('users').insert([payload]);
```

### In Triggers (Recommended for Shared Rules)

```cube
@beforeAdd({
    name: "validate_user";
    description: "Centralized user validation";
    function: ({newData}) => {
        if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(newData.email)) {
            throw new Error('Invalid email format');
        }
    };
});
```

The rule then applies to **every** insert through Dbcube, no matter which service performs it.

## Best Practices

1. **Be specific in WHERE clauses** — prefer primary keys or unique columns when targeting single rows.
2. **Verify scope before mass writes** — run the same conditions through `count()` first:

```typescript
const affected = await db.table('users')
  .where('last_login', '<', cutoff)
  .count();

if (affected < 1000) {
  await db.table('users').where('last_login', '<', cutoff).update({ status: 'dormant' });
}
```

3. **Batch your inserts** — one `insert()` with 50 objects beats 50 calls with one object.
4. **Wrap writes in try/catch** — engine errors arrive formatted with the offending code location.
