## JOIN Types

Dbcube provides three join methods, all with the same signature: `(table, column1, operator, column2)`.

| Method | SQL | Returns |
|--------|-----|---------|
| `join()` | INNER JOIN | Only rows with a match in **both** tables |
| `leftJoin()` | LEFT JOIN | All rows from the base table + matches (or NULLs) |
| `rightJoin()` | RIGHT JOIN | All rows from the joined table + matches (or NULLs) |

### join() — INNER JOIN

```typescript
const orders = await db.table('orders')
  .select(['orders.id', 'orders.total', 'users.name'])
  .join('users', 'orders.user_id', '=', 'users.id')
  .get();
// Only orders that have a matching user
```

### leftJoin() — LEFT JOIN

```typescript
const users = await db.table('users')
  .select(['users.id', 'users.name', 'orders.id'])
  .leftJoin('orders', 'users.id', '=', 'orders.user_id')
  .get();
// Every user — order columns are null for users without orders
```

### rightJoin() — RIGHT JOIN

```typescript
const orders = await db.table('users')
  .select(['users.name', 'orders.id', 'orders.total'])
  .rightJoin('orders', 'users.id', '=', 'orders.user_id')
  .get();
// Every order — user columns are null for orphaned orders
```

::callout{type="info"}
**Qualify your columns.** In joined queries always prefix columns with the table name (`users.id`, `orders.total`) to avoid ambiguity errors.
::

## Multiple JOINs

Chain join calls to traverse several relationships:

```typescript
const lines = await db.table('order_items')
  .select([
    'order_items.quantity',
    'products.name',
    'orders.id',
    'users.email'
  ])
  .join('orders', 'order_items.order_id', '=', 'orders.id')
  .join('products', 'order_items.product_id', '=', 'products.id')
  .join('users', 'orders.user_id', '=', 'users.id')
  .where('orders.status', '=', 'paid')
  .get();
```

## JOINs + Conditions

All `where` methods work on joined columns:

```typescript
const premiumOrders = await db.table('orders')
  .select(['orders.id', 'orders.total', 'users.name'])
  .join('users', 'orders.user_id', '=', 'users.id')
  .where('users.subscription', '=', 'premium')
  .whereBetween('orders.total', [100, 10000])
  .orderBy('orders.total', 'DESC')
  .get();
```

### Finding Orphans with leftJoin + whereNull

The classic "users without orders" pattern:

```typescript
const usersWithoutOrders = await db.table('users')
  .select(['users.id', 'users.name'])
  .leftJoin('orders', 'users.id', '=', 'orders.user_id')
  .whereNull('orders.id')
  .get();
```

## JOINs + groupBy

Combine joins with grouping for per-entity listings:

```typescript
const activeBuyers = await db.table('users')
  .select(['users.id', 'users.name'])
  .join('orders', 'users.id', '=', 'orders.user_id')
  .where('orders.status', '=', 'paid')
  .groupBy('users.id')
  .orderBy('users.name', 'ASC')
  .get();
// One row per user that has at least one paid order
```

::callout{type="warning"}
**Aggregated values per group** (e.g. "order count per user") aren't expressible with the aggregation methods, which return a single number for the whole query. For per-group metrics, fetch the grouped rows and aggregate in JavaScript, or model the metric as a column maintained by [triggers](/guides/query-builder/runtime-triggers).
::

## Defining the Relationships

JOINs work best when the schema declares the foreign keys — Dbcube then validates them and orders migrations correctly:

```ts
// orders.table.cube
@database("myapp");

@meta({
    name: "orders";
    description: "Customer orders";
});

@columns({
    id: {
        type: "int";
        options: ["primary", "autoincrement"];
    };
    user_id: {
        type: "int";
        options: ["not null", "index"];
        foreign: {
            table: "users";
            column: "id";
        };
    };
    total: {
        type: "decimal";
        length: "10,2";
        options: ["not null"];
    };
});
```

## Real-World Examples

### Blog: Posts with Author and Category

```typescript
const posts = await db.table('posts')
  .select([
    'posts.id',
    'posts.title',
    'posts.published_at',
    'users.name',
    'categories.name'
  ])
  .join('users', 'posts.author_id', '=', 'users.id')
  .leftJoin('categories', 'posts.category_id', '=', 'categories.id')
  .where('posts.status', '=', 'published')
  .orderBy('posts.published_at', 'DESC')
  .limit(10)
  .get();
```

### E-commerce: Order Detail View

```typescript
const detail = await db.table('order_items')
  .select([
    'products.name',
    'products.sku',
    'order_items.quantity',
    'order_items.unit_price'
  ])
  .join('products', 'order_items.product_id', '=', 'products.id')
  .where('order_items.order_id', '=', orderId)
  .orderBy('products.name', 'ASC')
  .get();
```

### Support: Tickets with Assigned Agent (Including Unassigned)

```typescript
const tickets = await db.table('tickets')
  .select(['tickets.id', 'tickets.subject', 'agents.name'])
  .leftJoin('agents', 'tickets.agent_id', '=', 'agents.id')
  .where('tickets.status', '=', 'open')
  .orderBy('tickets.created_at', 'ASC')
  .get();
// agents.name is null for unassigned tickets
```

## Performance Tips

1. **Index your join columns** — declare `"index"` in the foreign key column options in your `.cube` files.
2. **Select only the columns you need** — joined queries multiply row width quickly.
3. **Filter early** — conditions on the base table reduce the rows the join must process.
4. **Prefer `join()` over `leftJoin()`** when you only want matched rows; it's less work for the engine.
