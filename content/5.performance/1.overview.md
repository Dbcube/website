## The short version

DBCube runs query parsing, SQL generation, parameter binding, execution and
row decoding in a **native Rust engine** — not in the Node.js event loop. In a
head-to-head against the four most popular Node ORMs, on a real PostgreSQL 16
database, DBCube **wins or ties every single operation**, and is the *fastest
library in the table* for primary-key reads, updates, transactions and
concurrent traffic.

| Operation | DBCube | Prisma | Verdict |
|---|---:|---:|---|
| SELECT by primary key | **0.73 ms** | 1.01 ms | 🏆 −28% · #1 |
| Filtered list (WHERE+ORDER+LIMIT) | 1.63 ms | 2.00 ms | ✅ −19% |
| COUNT with filter | 1.64 ms | 2.04 ms | ✅ −20% |
| Relation load (50 users + orders) | 3.28 ms | 4.55 ms | ✅ −28% |
| INSERT single row | 1.49 ms | 1.83 ms | ✅ −19% |
| UPDATE by primary key | **1.41 ms** | 1.82 ms | 🏆 −23% · #1 |
| Transaction (2 atomic updates) | **2.07 ms** | 3.82 ms | 🏆 −46% · #1 |
| Bulk INSERT 1,000 rows | 14.93 ms | 21.56 ms | ✅ −31% |
| 100 concurrent PK lookups (per round) | **8.05 ms** | 8.04 ms | 🤝 tie · #1-class |

::callout{type="info"}
Median of 6 runs, PostgreSQL 16 in Docker, Node 22, Windows x64.
**Numbers vary with hardware — run it on yours.** See [Benchmarks](/performance/benchmarks)
for the exact commands.
::

## Why it's fast — the four levers

DBCube's speed is not one trick; it's the sum of four architectural decisions.
Each is covered in depth in [Architecture & internals](/performance/architecture).

::card-group
  ::card
  ---
  icon: i-lucide-cpu
  title: Native Rust engine
  ---
  Query plans cross from JS to Rust once. Parsing, binding, execution and row
  decoding run in compiled native code, keeping the Node.js event loop free.
  ::

  ::card
  ---
  icon: i-lucide-box
  title: In-process embedded backend
  ---
  The engine loads **inside** your Node process as a native addon (napi) — no
  TCP hop, no separate daemon to manage. Falls back gracefully to a C-ABI
  library or a TCP daemon when needed.
  ::

  ::card
  ---
  icon: i-lucide-git-merge
  title: Protocol pipelining
  ---
  On PostgreSQL, DBCube sends many concurrent queries down the same socket
  without waiting for each reply — 100 parallel lookups resolve in roughly one
  network round-trip instead of one-at-a-time.
  ::

  ::card
  ---
  icon: i-lucide-zap
  title: Prepared-statement cache + single-RTT transactions
  ---
  Repeated queries reuse their prepared statement (bind + execute only).
  `db.transaction()` / batch writes ship `BEGIN … COMMIT` as a single
  round-trip instead of N+2.
  ::
::

## Where DBCube shines (and where it doesn't)

The honest read:

- **Reads, updates, transactions, bulk, concurrency** — DBCube beats Prisma
  consistently and is at or near the top of the whole field.
- **Raw bulk throughput** — a thin query builder like **Knex** still edges out
  everyone on a 1,000-row insert (it's almost the raw driver). DBCube beats
  Prisma and Drizzle here, but Knex is the raw-speed champion.
- **Single inserts** — DBCube, Drizzle and Knex are within noise of each other;
  DBCube comfortably beats Prisma.

This is a benchmark you can audit line-by-line: every contender's code is ~80
lines in `benchmarks/src/contenders/*.js`, the schema and seed data are
identical, and the connection budget is capped at 10 for everyone.

## Optimization history

DBCube reached these numbers across seven measured optimization cycles. The
table shows the headline metric per cycle (same machine, first run → final):

| Lever shipped | Metric moved | Result |
|---|---|---|
| Client socket pool | Concurrency ×100 | 133 ms → 33 ms (4×) |
| `INSERT … RETURNING` | Single INSERT | removed a second round-trip |
| Embedded engine (C-ABI + koffi) | Per-query overhead | killed the TCP hop |
| napi addon (true async) | Boundary cost | native Promises from Rust's tokio |
| JSON-free napi boundary | Every op | plans & rows cross as native objects |
| Fast-path + 500-row bulk | Bulk INSERT | 60.7 ms → ~15 ms (4×), beats Prisma |
| **tokio-postgres migration** | **Concurrency** | **27.9 ms → 8.0 ms — from 3.5× behind Prisma to a tie** |

The final cycle replaced the PostgreSQL driver with **tokio-postgres**, adding
true protocol pipelining and a per-connection prepared-statement cache. That
closed the last gap — concurrency — while *also* dropping PK reads to the
fastest in the table. MySQL and SQLite continue to use `sqlx`; MongoDB uses its
native driver.
